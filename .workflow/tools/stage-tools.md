# Stage 工具约束与推荐

## 工具能力类型定义

为降低新增工具的维护成本，所有 catalog 工具按能力类型分类：

| 能力类型 | 说明 | 典型工具 |
|---------|------|---------|
| **读写型** | 直接读取或修改文件内容 | Read, Write, Edit, NotebookEdit |
| **执行型** | 运行系统命令、编译、测试、Git 操作 | Bash |
| **搜索型** | 查找文件、搜索内容、网络检索 | Grep, Glob, WebSearch |
| **协调型** | 派发 subagent、调用 skill | Agent, Skill |
| **上下文管理型** | 压缩、清空上下文或新开 agent | `/compact`, `/clear`, `/new` |

## 新增工具默认规则

- 只要工具已在 `catalog/` 中定义并在 `keywords.yaml` 中注册，且其**能力类型**未落入当前 stage 的**禁止类型**，即**默认可用**
- 各 stage 下方的"推荐工具"仅表示该 stage 的高频/高优先级工具，不构成白名单
- 若新增工具的能力类型不明确，默认参考其 catalog 文档中的"适用场景"进行归类

## requirement_review / planning

### 禁止的能力类型
- **执行型**：不得执行任何系统命令（如 `Bash`）
- **读写型（项目代码文件）**：不得使用 `Write`/`Edit` 修改项目业务代码或配置文件
- **例外**：允许使用 `Read` 读取文档、允许使用 `Write` 创建工作流文档（如 `requirement.md`、`change.md`）

### 推荐工具
- **Read**：读取现有文档、状态文件、角色文件
- **Agent**：派发分析类 subagent 进行需求澄清或边界讨论

### 上下文管理
- **允许**：`/compact`（当上下文负载达到预警阈值时）
- **限制**：避免使用 `/clear`，除非任务刚开始且上下文混乱
- **注意**：确保关键决策已记录到文件再执行维护

## executing

### 禁止的能力类型
- 无额外禁止（遵循 Harness 通用安全原则即可）

### 推荐工具
- **Bash**：编译、运行测试、Git 操作
- **Edit / Write**：按 plan.md 步骤修改代码、创建文件
- **Grep / Glob**：查找代码位置、搜索依赖关系
- **Agent**：执行复杂子任务时派发 subagent

### 上下文管理
- **允许**：`/compact`（主要维护动作）、`/clear`（仅任务刚开始时）
- **触发条件**：根据 chg-01 阈值和 chg-02 决策树
- **最佳实践**：
  1. 定期检查上下文负载（阶段转换时、任务完成时）
  2. 达到预警阈值（70-80%）时建议 `/compact`
  3. 达到强制维护阈值（85-90%）时必须执行维护
  4. 任务进行中时优先使用 `/compact`
  5. 任务刚开始时可考虑 `/clear`

## testing

### 禁止的能力类型
- **读写型**：不得使用 `Write`/`Edit`/`NotebookEdit` 修改被测内容（避免污染测试对象）
- **例外**：允许修改测试用例文件、测试脚本、文档类文件

### 推荐工具
- **Bash**：运行测试命令、查看测试报告
- **Read**：读取测试日志、被测代码
- **Grep / Glob**：在测试结果中定位错误信息
- **Agent**：派发测试评估 subagent

### 上下文管理
- **允许**：`/compact`、`/clear`
- **适用场景**：
  1. 从 executing 阶段转换到 testing 阶段时，建议执行 `/compact`
  2. 测试环境混乱时，可考虑 `/clear` 重新开始
  3. 测试用例过多导致上下文过长时，使用 `/compact`
- **限制**：确保测试状态和结果已记录到文件

---

## acceptance

### 禁止的能力类型
- **读写型**：不得使用 `Write`/`Edit`/`NotebookEdit`（只读核查）
- **执行型**：不得使用 `Bash`（避免在验收阶段引入环境变更）

### 推荐工具
- **Read**：逐条核对需求文档、变更文档、测试报告
- **Grep / Glob**：在代码库中查找待验收的实现细节
- **Agent**：派发验收评估 subagent

### 上下文管理
- **允许**：`/compact`
- **限制**：避免使用 `/clear`，除非验收完全失败需要重新开始
- **注意**：验收阶段通常上下文负载较低，维护需求较少

---

## regression

### 禁止的能力类型
- **读写型**：在确认问题前不得使用 `Write`/`Edit`/`NotebookEdit` 进行修复
- **执行型**：仅允许**只读命令**（如查看日志、查状态），不得使用 `Bash` 修改环境或部署

### 推荐工具
- **Grep / Glob**：在代码和日志中定位问题根因
- **Read**：阅读相关代码、配置、历史变更
- **Bash（只读）**：查看系统日志、运行状态检查命令
- **Agent**：派发诊断类 subagent 进行独立分析

### 上下文管理
- **允许**：新开 agent（主要）、`/compact`、`/clear`
- **适用场景**：
  1. **复杂问题诊断**：上下文已爆满时，必须新开 agent
  2. **历史分析**：需要大量历史日志分析时，使用 `/compact`
  3. **环境清理**：诊断环境混乱时，使用 `/clear`
- **特殊要求**：新开 agent 时必须创建完整的 `handoff.md` 交接文件（遵循 chg-03 协议）

## 上下文管理工具通用规则

### 1. 触发条件（引用 chg-01）
- **预警阈值**（70-80% 最大上下文）：建议执行维护
- **强制维护阈值**（85-90% 最大上下文）：必须执行维护
- **紧急阈值**（>95% 最大上下文）：必须新开 agent

### 2. 决策逻辑（引用 chg-02）
- **任务状态考虑**：任务刚开始、进行中、接近完成、已完成
- **关键决策记录**：确保重要信息已保存到文件
- **阶段适配**：不同阶段有不同的维护策略

### 3. 执行要求
1. **执行前检查**：确认关键信息已保存
2. **执行后验证**：验证维护效果，检查关键信息保留
3. **经验记录**：在 `context/experience/tool/claude-code-context.md` 中记录维护经验

### 4. 风险控制
- **`/clear` 风险**：所有历史信息丢失，需谨慎使用
- **`/compact` 风险**：可能丢失细节，需验证关键信息
- **新开 agent 风险**：需要完整交接，否则任务中断

### 5. 阶段特定建议
| Stage | 推荐动作 | 限制 | 注意事项 |
|-------|----------|------|----------|
| **planning/executing** | `/compact` | 避免 `/clear`（除非任务刚开始） | 保留关键决策和任务状态 |
| **testing** | `/compact` 或 `/clear` | 根据测试需求选择 | 确保测试状态已记录 |
| **acceptance** | `/compact` | 避免 `/clear` | 通常不需要频繁维护 |
| **regression** | 新开 agent | 复杂问题必须交接 | 遵循 chg-03 handoff 协议 |
