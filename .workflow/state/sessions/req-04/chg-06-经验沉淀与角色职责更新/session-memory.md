#### 1.1 收集本次上下文爆炸的完整信息

**现象**：
- 用户收到 API 错误：`This model's maximum context length is 102400 tokens. However, you requested 102838 tokens`
- 上下文长度超出 Claude API 限制（102400 tokens）
- 工作流完全中断，无法继续执行任务

**根因**：
1. **缺乏监控机制**：没有定期检查上下文负载的机制
2. **缺乏阈值定义**：没有明确的预警、强制维护、紧急阈值
3. **缺乏维护策略**：没有定义何时使用 `/compact`、`/clear` 或新开 agent
4. **缺乏交接协议**：没有标准化的 handoff 交接文件模板
5. **职责不明确**：各角色在上下文维护中的职责不清晰

**影响**：
1. **工作流中断**：当前任务无法继续，需要重新开始
2. **时间浪费**：已完成的进度丢失，需要重复工作
3. **用户挫败感**：体验差，影响工作效率
4. **数据丢失风险**：未保存的决策和部分完成工作可能丢失

**解决方案**：
1. **chg-01**：设计监控指标与阈值（预警70-80%、强制维护85-90%、紧急>95%）
2. **chg-02**：设计维护动作决策树（基于阈值和任务状态选择动作）
3. **chg-03**：设计 handoff 交接协议（标准化的交接文件模板）
4. **chg-04**：工具层集成（Claude Code 上下文管理工具集成）
5. **chg-05**：风险与恢复路径（风险注册和恢复流程）

**效果**：
- 建立了完整的上下文维护机制
- 提供了明确的决策支持
- 标准化了交接流程
- 注册了风险并定义了恢复路径

#### 1.2 识别可泛化的教训

**本次特有的**：
- 具体触发 req-03 regression 诊断的场景
- 具体的 API 错误消息格式
- 本次需求（req-04）的完整设计过程

**可能在其他场景重复发生的**：
1. **上下文监控缺失**：任何长时间会话都可能积累上下文
2. **阈值触发机制**：所有使用 Claude Code 的工作流都需要阈值定义
3. **维护决策困难**：用户不知道何时使用 `/compact`、`/clear` 或新开 agent
4. **交接不完整**：新开 agent 时信息传递不完整
5. **职责模糊**：各角色在维护中的职责不明确

**需要纳入标准流程的**：
1. **定期监控**：主 agent 应定期检查上下文负载
2. **阈值触发**：达到阈值时必须触发维护动作
3. **决策支持**：提供明确的决策树支持维护选择
4. **标准化交接**：新开 agent 必须使用标准 handoff 协议
5. **经验沉淀**：每次维护后记录经验教训
6. **角色职责**：明确各角色在上下文维护中的职责
