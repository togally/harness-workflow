## 常见问题与解决方案

### 问题一：`/compact` 后关键信息丢失

#### 症状
- 压缩后关键决策或任务状态信息丢失
- 需要重新读取文件或重新建立上下文
- 影响任务连续性

#### 原因
1. 压缩算法可能过度压缩重要信息
2. 未在压缩前确认关键信息已保存到文件
3. 重新加载文件不完整

#### 解决方案
1. **压缩前检查**：
   - 确认所有关键决策已记录到 `session-memory.md`
   - 验证任务状态已保存
   - 检查必要文件已更新

2. **标准化重新加载清单**：
   - 创建标准化的重新加载文件清单
   - 包括：runtime.yaml、session-memory.md、当前变更文档、相关配置文件
   - 确保清单覆盖所有必要信息

3. **验证机制**：
   - 压缩后立即验证关键信息是否保留
   - 如有遗漏，补充读取相关文件
   - 记录验证结果和经验

### 问题二：`/clear` 导致历史信息完全丢失

#### 症状
- 所有历史对话信息丢失
- 需要重新加载大量文件
- 可能遗漏未保存的重要信息

#### 原因
1. 未在执行前保存所有重要信息
2. 对 `/clear` 的风险认识不足
3. 在不适用的场景使用 `/clear`

#### 解决方案
1. **严格适用场景**：
   - 仅用于任务刚开始或已完成时
   - 避免在任务进行中使用
   - 确保历史上下文已无效或干扰

2. **执行前检查清单**：
   - 关键决策已记录到文件
   - 任务状态已保存到 `session-memory.md`
   - 所有必要文件已更新
   - 确认没有未保存的重要信息

3. **风险提示**：
   - 明确告知用户 `/clear` 的风险
   - 提供替代方案（如 `/compact`）
   - 记录使用经验和教训

### 问题三：新开 agent 交接不完整

#### 症状
- handoff 文件缺失重要信息
- 新会话无法继续任务
- 需要重新建立上下文

#### 原因
1. handoff 文件不完整
2. 未遵循 chg-03 handoff 协议
3. 重要文件未传递

#### 解决方案
1. **遵循标准协议**：
   - 严格遵循 chg-03 的 `handoff.md` 模板
   - 包含所有必填字段
   - 验证文件完整性

2. **完整文件传递**：
   - `session-memory.md`
   - 当前变更的 `change.md`、`plan.md`、`design.md`
   - 相关配置文件
   - 生成的代码或文档

3. **接收方指引**：
   - 明确告知如何加载交接内容
   - 提供下一步动作指引
   - 注意事项和风险提示

- **需求/变更：** req-03/regression → req-04/chg-01~chg-06

- **时间：** 2026-04-14（整个 req-04 设计周期）

### 上下文状态
- **Token 估算：** >100%（102838 tokens，超出限制 438 tokens）
- **消息条数：** 未记录（推测 >200 条）
- **文件读取次数：** 未记录（推测 >100 次）
- **会话时长：** 未记录（推测 >4 小时）
- **工具调用分布：** 未记录

### 决策过程
- **阈值判断：** 紧急阈值（>95%，实际 >100%）
- **任务状态：** 进行中（req-03 done 阶段回顾）
- **关键决策记录：** 部分记录（部分决策在会话中，未保存到文件）
- **选择理由：** 上下文已爆满，API 报错，必须新开 agent
- **引用决策树：** chg-02 决策表：紧急阈值 → 必须新开 agent

### 执行过程
- **维护动作：** 新开 agent（手动，无标准化交接）
- **执行前检查：** 未系统化检查，部分信息丢失

### 效果评估
- **维护后上下文负载：** 重新开始，负载正常
- **关键信息保留：** 部分丢失，需要重新回忆和重建
- **任务连续性：** 中断，需要重新开始任务
- **性能改善：** 新会话响应正常

### 经验教训
- **成功做法：** 
  1. 触发 regression 诊断，系统化分析问题
  2. 创建 req-04 专门解决上下文维护问题
  3. 设计完整的六层解决方案

  1. 缺乏监控机制，直到 API 报错才发现问题
  2. 缺乏阈值定义，不知道何时需要维护
  3. 缺乏决策支持，不知道选择哪种维护动作
  4. 缺乏交接协议，新开 agent 时信息传递不完整
  5. 缺乏角色职责，各角色在维护中的职责不明确
- **改进建议：**
  1. **预防优于恢复**：建立定期监控机制，避免达到紧急阈值
  2. **标准化决策**：使用 chg-02 决策树提供明确的决策支持
  3. **标准化交接**：新开 agent 必须使用 chg-03 handoff 协议
  4. **工具层集成**：在 tools 层集成 Claude Code 上下文管理工具
  5. **风险注册**：在 constraints 层注册上下文爆炸风险
  6. **经验沉淀**：记录每次维护经验，持续优化
  7. **角色职责**：明确各角色在上下文维护中的职责
- **风险提示：**
  1. **监控盲区**：不监控就会在 API 报错时才发现问题
  2. **决策困难**：没有决策支持会导致选择错误的维护动作
  3. **交接不完整**：没有标准化交接会导致信息丢失
  4. **职责模糊**：职责不明确会导致无人负责维护

### 验证结果
- **任务是否继续：** 是，通过 req-04 设计完整的上下文维护机制
- **后续问题：** 无，req-04 提供了完整的解决方案
- **用户反馈：** 通过系统化设计解决了根本问题

## 从本次教训中提炼的最佳实践

### 1. 建立完整的上下文维护生命周期
**问题**：临时应对，缺乏系统性
**解决方案**：设计完整的六层解决方案：
- **监控层**（chg-01）：定义指标、阈值、采集方法
- **决策层**（chg-02）：基于阈值和任务状态的决策树
- **交接层**（chg-03）：标准化的 handoff 交接协议
- **工具层**（chg-04）：Claude Code 工具集成
- **风险层**（chg-05）：风险注册和恢复路径
- **经验层**（chg-06）：经验沉淀和角色职责更新

### 2. 预防性监控优于被动恢复
**问题**：等到 API 报错才处理
**解决方案**：建立三层阈值预警机制：
- **预警阈值**（70-80%）：提醒关注，建议维护
- **强制维护阈值**（85-90%）：必须执行维护
- **紧急阈值**（>95%）：立即处理，优先新开 agent

### 3. 标准化决策支持
**问题**：用户不知道何时使用哪种维护动作
**解决方案**：chg-02 决策树提供明确的决策逻辑：
- 基于当前负载水平（阈值）
- 基于任务状态（刚开始、进行中、接近完成、已完成）
- 基于关键决策记录情况
- 输出明确的维护动作建议

### 4. 确保交接完整性
**问题**：新开 agent 时信息丢失
**解决方案**：chg-03 handoff 协议确保完整交接：
- 标准化的模板结构
- 必填字段确保关键信息不遗漏
- 接收方指引确保顺利继续
- 完整性验证确保交接成功

### 5. 明确角色职责
**问题**：各角色在维护中的职责不明确
**解决方案**：chg-06 角色职责更新：
- **主 agent**：监控、触发、协调、交接管理
- **subagent**：消耗报告、清理建议、状态沟通
- **用户**：关注提醒、确认动作、提供信息

### 6. 持续经验沉淀
**问题**：重复犯同样的错误
**解决方案**：建立经验沉淀机制：
- 每次维护后记录经验
- 分类整理便于检索
- 用于决策参考和流程优化
- 持续改进维护实践
