# Stage 角色模板

本文件是 Harness Workflow 中各 stage 角色文件的统一结构模板。所有 stage 角色文件（包括 requirement-review、planning、executing、testing、acceptance、regression、done、tools-manager）应遵循以下章节顺序和格式。

---

## 核心原则

**每个角色必须包含且仅包含两类流程定义：**

1. **角色的工作流程（Role Lifecycle）**：角色被加载后，作为 agent 自身如何执行、如何收尾、如何交接的生命周期。这是角色的核心执行指南，必须完整、无歧义。
2. **Stage 的业务流程（Stage SOP）**：该 stage 特有的业务操作步骤。两者通常合并表述在"标准工作流程（SOP）"章节中，但必须覆盖角色的完整生命周期（初始化 → 执行 → 退出 → 交接）。

---

## 硬门禁：继承 base-role

**创建任何子角色时，必须首先加载并继承 base-role.md。base-role 中定义的规则自动适用于所有子角色，无需重复声明。**

**禁止重复声明以下内容（已在 base-role 中定义）：**
- 硬门禁一（工具优先委派）
- 硬门禁二（操作说明与日志）
- 硬门禁三（角色自我介绍）
- 上下文维护规则（70%/85% 阈值）
- 经验沉淀规则

**子角色 Step 0 格式**：只需写业务特有的初始化内容
```markdown
### Step 0: 初始化
- 确认前置上下文已加载（runtime.yaml、base-role.md、stage-role.md、本角色文件）
- [业务特有的初始化内容]
```

---

## 章节顺序

```
1. 角色定义
2. 标准工作流程（SOP）—— 必须覆盖角色完整生命周期
3. 可用工具
4. 允许的行为
5. 禁止的行为
6. 上下文维护职责
7. 职责外问题
8. 退出条件
9. ff 模式说明
10. 流转规则
11. 完成前必须检查
```

---

## 各章节说明

### 1. 角色定义
- 一句话定义你是谁
- 明确本 stage 的核心任务和边界
- 说明你是主 agent 的编排对象还是辅助角色

### 2. 标准工作流程（SOP）
**本章节是角色文件的核心，必须存在且不可为空。**

- 使用 `### Step X: 步骤名称` 格式
- 每个步骤应包含可执行的检查点（bullet list）
- 必须覆盖角色的完整生命周期：
  - **初始化**：被加载后第一步做什么（读取哪些文件、建立什么上下文）
  - **执行**：本 stage 的核心业务步骤
  - **退出**：如何判断任务已完成
  - **交接**：如何保存状态、如何向主 agent 报告结果
- 对于涉及上下文交接的 stage，应在 Step 1 明确 session start 时的加载要求
- 对于涉及 stage 切换的，应在最后一步明确退出和交接动作

### 3. 可用工具
- 引用 `.workflow/tools/stage-tools.md#{stage}`
- 若可用工具不受白名单限制（如 done 阶段），需明确说明

### 4. 允许的行为
- bullet list 列出本 stage 允许做的事

### 5. 禁止的行为
- bullet list 列出本 stage 绝对不能做的事
- 通常包含"不得修改代码"、"不得跳过步骤"等

### 6. 上下文维护职责
- 消耗报告
- 清理建议
- 状态保存

### 7. 职责外问题
统一格式：
```markdown
遇到职责范围外的问题，不自行处理，记录并上报给主 agent。规则见 `.workflow/constraints/boundaries.md#职责外问题处理规则`。
```

### 8. 退出条件
- 使用 `- [ ]` 检查清单格式
- 条件应可验证、无歧义
- 主 agent 根据退出条件判断是否可以推进到下一 stage

### 9. ff 模式说明
- 说明 ff 模式下该 stage 的自动完成判定条件
- 说明完成后主 agent 的自动推进行为
- 若本 stage 不支持 ff 模式，明确说明

### 10. 流转规则
- 明确 `harness next` 的流转方向
- 明确 `harness regression` 的触发条件
- 说明 ff 模式下的自动流转行为

### 11. 完成前必须检查
- 使用编号列表（1. 2. 3.）
- 问题应能暴露常见错误
- 最后一项通常为"关键决策和进度是否已保存到 session-memory？"

---

## 特殊说明

- **done 阶段**：由于 done 由主 agent（技术总监）亲自执行，可在 SOP 之后保留原有的六层检查清单、工具层适配性问题模板等特殊章节，但总体结构仍应遵循本模板的章节顺序。
- **tools-manager**：作为辅助角色，同样应遵循本模板结构，确保行为边界完整。
- **技术总监（Director）**：虽然不属于 stage 角色，但同样应遵循本模板的核心原则——必须有明确的角色生命周期流程定义。
