# 风险识别规则

**触发时机：** before-task hook，每个任务开始前自动扫描

## 高风险关键词

| 类别 | 关键词 |
|------|--------|
| 数据破坏 | 删除数据、清空、drop、truncate、rm -rf |
| 生产环境 | production、prod、线上、发布、release |
| 大范围变更 | 全量替换、重构所有、删除所有、migrate |
| 权限/安全 | sudo、chmod 777、credentials、密钥、token |
| 不可逆 Git | force push、reset --hard、branch -D |

## 风险处理规则

```
识别到高风险关键词
    ↓
评估两个维度：可逆性 + 影响范围

可逆 + 范围小（<5个文件，可 git 回滚）
    → 记录到 session-memory，继续执行

不可逆 或 范围大
    → 暂停执行
    → 向用户说明：操作内容、风险点、建议替代方案
    → 等待用户明确确认后再继续

涉及生产/安全/权限
    → 强制停止
    → 必须人工授权，不得自行判断
```

## 快速判断原则
- 能 git 回滚的 → 低风险
- 影响范围只在 .workflow/ 内的 → 中风险（说明后继续）
- 影响项目代码或生产的 → 高风险（必须确认）

## 上下文爆炸风险

### 风险条目：上下文爆炸导致工作流中断

#### 风险描述
- **触发条件**：上下文长度接近或超过 Claude API 最大限制（102400 tokens）
- **具体表现**：
  - API 返回 400/413 错误："context length exceeded"
  - 所有工具调用失败，工作流完全中断
  - 当前会话无法继续，必须重新开始
- **影响范围**：整个工作流会话，所有未保存的进度和决策

#### 影响分析
1. **会话丢失**：当前对话历史全部丢失，无法恢复
2. **任务中断**：进行中的任务被迫中止，进度丢失
3. **重新开始成本**：需要重新加载文件、重建上下文、重新执行已完成步骤
4. **数据不一致风险**：部分文件可能已更新，但相关上下文丢失导致状态不一致
5. **用户体验**：用户挫败感增加，对系统可靠性产生怀疑

#### 缓解措施
1. **监控预警**：引用 chg-01 设计的监控指标和阈值
   - 预警阈值（70-80%）：提醒用户关注上下文负载
   - 强制维护阈值（85-90%）：必须执行维护动作
   - 紧急阈值（>95%）：立即处理，优先考虑新开 agent
2. **定期维护**：根据 chg-02 决策树执行维护动作
   - `/compact`：压缩历史消息，保留关键信息
   - `/clear`：清空历史，重新开始
   - 新开 agent：交接任务，在新会话中继续
3. **handoff 准备**：引用 chg-03 handoff 协议，确保紧急情况下能顺利交接

#### 检测方法
1. **主动监控**：主 agent 定期估算上下文负载（Token 估算百分比）
2. **阈值触发**：当达到 chg-01 定义的任一阈值时触发风险检测
3. **性能指标**：消息条数 >200、文件读取次数 >100、会话时长 >6 小时
4. **API 错误**：捕获 "context length exceeded" 等 API 错误作为事后检测

#### 风险等级
- **低风险**：<70% 上下文使用，正常监控
- **中风险**：70-85% 上下文使用，需要关注和准备维护
- **高风险**：85-95% 上下文使用，必须立即执行维护动作
- **紧急风险**：>95% 上下文使用，随时可能爆炸，需紧急处理

#### 处理规则
```
检测到上下文爆炸风险
    ↓
根据当前负载判断风险等级
    ↓
按 chg-02 决策树选择维护动作
    ├── 预警阈值 → 建议 `/compact`
    ├── 强制维护阈值 → 必须执行维护（根据任务状态选择动作）
    └── 紧急阈值 → 必须新开 agent + handoff 交接
    ↓
执行维护动作，验证恢复效果
    ↓
记录到 session-memory，用于经验沉淀
```
