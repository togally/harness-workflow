# 行为边界

## 文件操作边界
- 只操作 `.workflow/` 和项目代码，不操作系统文件
- 删除操作（>5个文件）必须在 session-memory 记录原因
- 大范围删除前必须确认无法通过其他方式实现目标

## 跨层边界
- `.workflow/flow/` 只存工作文档（requirement/change/plan），不存状态数据
- `.workflow/state/` 管理所有状态，session-memory 不散落在 `.workflow/flow/` 里
- 角色文件不写业务规则，规则归对应层管理
- 主 agent 不直接执行节点任务，节点任务归 subagent

## 阶段边界
- 当前 stage 的禁止行为以角色文件为准
- 不得跳过 planning 直接进入 executing
- `done` 状态的需求不得被重新打开（只能新建需求）
- testing / acceptance 阶段不得修改被测内容

## ff 模式下 AI 自主决策边界

### AI 可以自行决定
- 文档内容的补充和细化
- 拆分规则的细化（在需求范围内）
- 实现方案的选择（在 plan 范围内）
- 测试策略和用例设计
- 代码级的小范围调整
- 遇到 skill 缺失时，按 `constraints/recovery.md#skill 缺失处理路径` 搜索替代方案

### AI 必须暂停 ff 模式并向用户求援
- 涉及外部系统凭据、密钥或权限
- 可能破坏生产环境或导致数据丢失
- 用户意图不明确或与已有需求冲突
- 超出 plan 范围的大架构改动
- regression 诊断后仍无法自动恢复
- 连续遇到平台级错误（如 API Error 400）且 `constraints/recovery.md` 中的恢复步骤失效

## 状态边界 
- `state/runtime.yaml` 只存全局运行状态，不存业务数据
- `state/requirements/{id}.yaml` 只存该需求的执行状态
- 经验沉淀到 `context/experience/`（stage/, tool/, risk/），不存在 session-memory 里长期留存

## 职责外问题处理规则

**触发条件**（以下任意一种均触发）：
1. AI 在执行过程中主动识别到当前角色职责范围外的 bug、需求或风险
2. 用户在对话中口头提出任何问题、想法或不满（不论是否使用了正式命令）

**各角色行为**：
- 不自行处理，不忽略
- 将问题标记为"职责外"，交主 agent 决策
- 继续当前节点任务，不因上报而中断

**上报格式**（角色向主 agent 的简短记录）：
```
来源阶段: <stage>
来源: AI识别 / 用户口头
问题描述: <一句话>
处置状态: pending
```

**主 agent 决策规则**：
1. 接收上报后，记录到当前 session-memory 的 `## 待处理捕获问题` 区块
2. 在当前节点任务完成时（或用户触发下一个 harness 命令前），逐条询问用户处置意向：
   - **A. 升级为正式 regression**：触发 `harness regression "<问题>"` 流程
   - **B. 本次忽略**：从待处理列表移除，不再提醒
   - **C. 下次再说**：保留 pending，下次会话继续提示
3. 未经用户决策前，不得擅自升级或忽略任何捕获问题

## 上下文维护行为边界

### 主 agent 职责边界
1. **监控职责**：
   - 必须定期检查上下文负载（阶段转换时、任务完成时）
   - 必须使用 chg-01 定义的指标和阈值进行评估
   - 必须记录监控结果到 session-memory

2. **维护触发边界**：
   - 达到预警阈值时必须提醒用户关注上下文状态
   - 达到强制维护阈值时必须执行维护动作
   - 达到紧急阈值时必须立即处理，优先保障工作流连续性

3. **决策执行边界**：
   - 必须遵循 chg-02 决策树选择维护动作
   - 重要维护动作（如 `/clear`、新开 agent）需用户确认
   - 必须验证维护效果，确保工作流可继续

### subagent 职责边界
1. **上下文消耗报告**：
   - 任务结束时必须报告上下文消耗情况
   - 必须估算任务期间新增的消息条数、文件读取次数
   - 必须报告关键决策是否已记录到文件

2. **维护建议边界**：
   - 发现上下文负载过高时可建议维护，但不得擅自执行
   - 建议必须基于 chg-01 阈值和 chg-02 决策逻辑
   - 必须将建议记录到 session-memory 供主 agent 参考

3. **信息保存边界**：
   - 关键决策必须在任务过程中记录到文件
   - 不得依赖上下文记忆保存重要信息
   - 任务中断前必须确保状态信息已保存

### 用户职责边界
1. **状态关注职责**：
   - 应关注系统提供的上下文状态提醒
   - 应及时响应维护建议和确认请求
   - 可随时通过 `harness context-status` 命令检查状态

2. **维护确认边界**：
   - 重要维护动作需用户明确确认
   - 用户可拒绝或延迟非紧急维护建议
   - 用户可主动要求执行维护或交接

3. **信息保存提醒**：
   - 用户应提醒 agent 保存关键决策到文件
   - 用户应关注未保存信息的风险提示
   - 用户应在长时间任务中主动检查保存状态

### 工具使用边界
1. **`/compact` 命令使用边界**：
   - 适用场景：预警阈值、强制维护阈值（任务进行中/接近完成）
   - 使用前提：关键决策已记录到文件
   - 使用后必须：验证关键信息保留、重新读取必要文件
   - 禁止使用：关键决策未记录、任务刚开始且历史无关

2. **`/clear` 命令使用边界**：
   - 适用场景：强制维护阈值（任务刚开始/已完成）、需要全新上下文
   - 使用前提：所有重要信息已保存到文件
   - 使用后必须：重新加载所有必要文件、重新建立上下文
   - 禁止使用：有未保存的重要信息、复杂进行中的任务

3. **handoff 交接边界**：
   - 适用场景：紧急阈值、性能问题、用户请求
   - 必须遵循：chg-03 handoff 协议标准模板
   - 必须包含：完整任务状态、关键决策、必须文件列表
   - 交接前必须：验证交接完整性、通知用户
   - 接收方必须：按指引加载文件、验证准确性、继续工作

### 边界违反处理
1. **监控职责违反**：
   - 主 agent 未定期检查上下文负载 → 记录到 session-memory，提醒改进
   - 未达到阈值却执行维护 → 回滚操作，记录错误经验
   - 达到阈值未及时处理 → 立即补救，分析原因，优化监控

2. **维护动作违反**：
   - 未遵循决策树选择动作 → 评估影响，调整决策，记录教训
   - 重要维护未获用户确认 → 暂停执行，获取确认，记录违规
   - 维护后未验证效果 → 补充验证，确保工作流可继续

3. **信息保存违反**：
   - 关键决策未记录导致丢失 → 尝试恢复，记录严重错误
   - 过度依赖上下文记忆 → 强制保存到文件，优化工作习惯
   - 交接信息不完整 → 补充信息，优化交接流程

4. **工具滥用处理**：
   - 不当使用 `/compact` 导致信息丢失 → 尝试恢复，记录使用限制
   - 不当使用 `/clear` 导致工作丢失 → 严重错误，必须从 handoff 或备份恢复
   - handoff 交接不规范 → 补充信息，重新交接，优化模板

### 边界执行原则
1. **安全第一**：优先保障信息安全和任务连续性
2. **用户中心**：重要决策需用户参与和确认
3. **透明可查**：所有维护操作必须记录到 session-memory
4. **持续改进**：边界违反应分析原因，优化流程
5. **经验沉淀**：成功经验和失败教训应沉淀到经验文件
