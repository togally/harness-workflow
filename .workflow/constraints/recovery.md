# 失败恢复规则

## 失败三条路径

### 路径一：重试（同路径，修正错误）
**适用：** 方案可行，执行出错（命令参数错误、文件路径写错等）
```
1. 记录错误到 session-memory（失败路径记录）
2. 找出具体错误原因
3. 修正后重新执行该步骤
4. 不超过 3 次重试，超过则切路径
```

### 路径二：切路径（换方案）
**适用：** 当前方案不可行，需要换一种实现方式
```
1. 记录放弃原因到 session-memory
2. 回到 planning 讨论替代方案
3. 更新 plan.md，再进入执行
```

### 路径三：回滚（恢复稳定状态）
**适用：** 文件/状态已损坏，无法通过重试或切路径修复
```
1. 停止所有操作
2. 记录损坏范围到 session-memory
3. 通过 git 恢复到最近稳定状态（git stash / git checkout）
4. 在 session-memory 标记："已回滚，原因：{xxx}"
5. 触发 harness regression 评估是否需要转为新变更
```

## 上下文爆满 / 任务切换处理

**适用：** context 接近上限、需要切换到其他需求、会话被中断

```
Step 1  识别需要交接的时机
        - context 接近上限（对话轮次多、文件读取多）
        - 用户运行 harness use 切换需求
        - 任何原因导致的会话中断

Step 2  写 handoff.md
        路径：state/sessions/{req-id}/[chg-id/]handoff.md
        内容：
          ## 触发原因
          ## 当前状态（需求、阶段、当前变更）
          ## 已完成的步骤（✅ 列表）
          ## 正在进行（中断点，▶ 标记的步骤）
          ## 下一步
          ## 已知约束与失败路径
          ## 接管注意事项

Step 3  新 agent 启动
        → session-start hook 检测到 handoff.md
        → 读取 handoff.md 内容接管工作
        → 从中断点继续，不重复已完成步骤

Step 4  完成后清理
        → 工作完成后删除 handoff.md
        → session-memory 保留（历史记录）
```

---

## 恢复检测逻辑（session-start hook 执行）

```
读 session-memory.md
    ├── 发现 handoff.md 存在 → 优先读 handoff.md，按交接内容接管
    ├── 发现 ▶ 未完成标记（无 handoff.md）→ 从该步骤重新执行
    └── 全部 ✅ → 正常继续下一步
```

## 上下文超限恢复路径

### 1. 检测阶段
**触发条件**（满足任一条件即触发）：
1. **主动监控发现**：主 agent 定期检查发现达到 chg-01 定义的阈值
   - 预警阈值（70-80%）：黄色警报
   - 强制维护阈值（85-90%）：橙色警报
   - 紧急阈值（>95%）：红色警报
2. **性能指标触发**：
   - 消息条数 >200 条
   - 文件读取次数 >100 次
   - 会话时长 >6 小时
3. **API 错误捕获**：收到 "context length exceeded" 错误

**检测方法**：
- 主 agent 在阶段转换、任务完成时主动估算
- subagent 在任务结束时报告上下文消耗
- 用户可通过 `harness context-status` 命令手动检查

### 2. 决策阶段
**决策依据**：chg-02 维护动作决策树

**决策流程**：
```
检测到上下文超限
    ↓
判断当前阈值级别（预警/强制维护/紧急）
    ↓
评估任务状态（阶段、进度、关键决策记录）
    ↓
选择维护动作：
    ├── 预警阈值 → 建议 `/compact`
    ├── 强制维护阈值 → 必须执行维护
    │   ├── 任务刚开始 → `/clear`
    │   ├── 任务进行中 → `/compact`
    │   ├── 任务接近完成 → `/compact`
    │   └── 任务已完成 → `/clear`
    └── 紧急阈值 → 必须新开 agent + handoff
    ↓
提供操作指引和预期效果说明
```

### 3. 执行阶段

#### 3.1 `/compact` 操作步骤
**适用**：预警阈值、强制维护阈值（任务进行中/接近完成）
```
1. 确保关键决策已记录到文件
2. 执行 `/compact` 命令
3. 重新读取必要文件：
   - `.workflow/state/runtime.yaml`
   - 当前 `session-memory.md`
   - 当前变更的 `change.md`、`plan.md`、`design.md`
   - 相关配置和产出文件
4. 验证关键信息是否保留：
   - 当前 stage 和 requirement
   - 已完成的工作步骤
   - 关键决策记录
   - 待处理问题
5. 如有遗漏，补充读取相关文件
6. 记录维护操作到 session-memory
```

#### 3.2 `/clear` 操作步骤
**适用**：强制维护阈值（任务刚开始/已完成）、需要全新上下文
```
1. **关键信息保存**（必须步骤）：
   - 所有关键决策记录到文件
   - 任务状态保存到 `session-memory.md`
   - 产出文件已保存和提交
2. 执行 `/clear` 命令
3. **重新加载所有必要文件**：
   - `.workflow/state/runtime.yaml`
   - `.workflow/context/index.md`
   - 当前角色文件（如 `executing.md`）
   - 相关经验文件（stage/, tool/, risk/）
   - 当前变更的所有文档
   - 产出文件和配置
4. **重新建立上下文**：
   - 了解当前任务状态
   - 加载关键决策记录
   - 确认下一步动作
5. 记录维护操作到 session-memory
```

#### 3.3 新开 agent + handoff 操作步骤
**适用**：紧急阈值、性能问题、用户请求
```
1. **创建 handoff 交接文件**：
   - 使用 chg-03 标准模板创建 `handoff.md`
   - 包含完整任务状态、关键决策、必须文件列表
   - 提供接收方指引和注意事项
2. **验证交接完整性**：
   - 所有必填字段完整
   - 必须传递的文件可访问
   - 信息准确无误
3. **通知用户**：告知交接原因和下一步操作
4. **新 agent 启动**：
   - 新会话读取 `handoff.md`
   - 按指引加载所有必要文件
   - 验证交接内容完整性
   - 从交接点继续工作
5. **清理**：工作完成后删除 `handoff.md`，保留 `session-memory`
```

### 4. 验证阶段
**验证目标**：确保恢复成功，工作流可继续

**验证内容**：
1. **上下文状态验证**：
   - 维护后上下文负载显著降低
   - 工具调用恢复正常
   - 无 API 错误
2. **信息完整性验证**：
   - 关键决策和状态信息保留
   - 必要文件可正常读取
   - 任务进度准确
3. **工作流连续性验证**：
   - 可从维护点继续工作
   - 下一步动作明确
   - 无信息丢失导致的障碍

**验证方法**：
- 执行简单工具调用测试（如 `Read` 当前文件）
- 检查关键信息是否可访问
- 验证任务状态与维护前一致
- 用户确认可继续工作

### 5. 分层恢复策略

#### 5.1 轻度超限（预警阈值）
- **策略**：建议维护，可延迟执行
- **动作**：`/compact`
- **目标**：压缩历史，保留关键信息，继续当前任务
- **时间窗口**：24小时内执行

#### 5.2 中度超限（强制维护阈值）
- **策略**：必须维护，立即执行
- **动作**：根据任务状态选择 `/compact` 或 `/clear`
- **目标**：确保工作流可继续，避免爆炸
- **时间窗口**：立即执行，不得延迟

#### 5.3 严重超限（紧急阈值）
- **策略**：紧急处理，优先保障连续性
- **动作**：新开 agent + handoff 交接
- **目标**：避免工作流中断，确保任务继续
- **时间窗口**：立即处理，优先于其他任务

### 6. handoff 集成要求
**必须遵循 chg-03 handoff 协议**：
1. **模板标准化**：使用标准 `handoff.md` 模板
2. **字段完整性**：包含所有必填字段
3. **文件可访问性**：确保必须传递的文件可访问
4. **接收指引明确**：提供清晰的继续工作指引
5. **验证机制**：交接前验证完整性，接收后验证准确性

**handoff 触发场景**：
1. 紧急阈值触发
2. `/compact` 或 `/clear` 失败
3. 用户明确要求交接
4. 系统性能问题无法解决

### 7. 恢复失败处理
**场景**：维护动作执行失败或效果不佳

**处理流程**：
```
维护动作失败
    ↓
分析失败原因
    ├── 信息丢失过多 → 回退到 handoff 交接
    ├── 工具调用失败 → 尝试替代方案或新开 agent
    └── 用户不满意 → 根据用户要求调整
    ↓
选择替代恢复方案
    ↓
执行并验证效果
    ↓
记录失败经验和改进建议
```

### 8. 记录与经验沉淀
**必须记录到 session-memory**：
1. **维护决策**：触发原因、阈值级别、选择动作
2. **执行过程**：具体步骤、遇到的问题、解决方法
3. **验证结果**：恢复效果、信息保留情况、用户反馈
4. **改进建议**：阈值调整、决策优化、工具改进

**经验沉淀到 `.workflow/context/experience/`**：
1. **成功案例**：有效的恢复策略和操作
2. **失败教训**：需要避免的问题和错误
3. **最佳实践**：推荐的维护时机和方法
4. **工具使用**：`/compact`、`/clear`、handoff 的使用技巧
