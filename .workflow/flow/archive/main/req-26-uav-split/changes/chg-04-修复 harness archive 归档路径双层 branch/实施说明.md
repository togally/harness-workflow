# 实施说明：chg-04 修复 harness archive 归档路径双层 branch

## 实际做了什么
- Step 1：全局定位 archive 路径拼接点。Grep 确认实现集中在 `src/harness_workflow/workflow_helpers.py` 的 `archive_requirement()`（L4335）与 `resolve_archive_root()`（L4219）。Bug 根因：`resolve_archive_root` 的 primary 形态返回 `artifacts/{branch}/archive`（已含 branch 段），但 `archive_requirement` 不分形态地再拼 `folder = _get_git_branch(root)`，在 primary 路径下产出 `artifacts/main/archive/main/req-xx/`（双层 branch）；legacy 路径 `.workflow/flow/archive` 不含 branch，原来恰好单层，所以历史仓里看起来"正常"。
- Step 2：修 `archive_requirement()`（新实现 L4335-L4370）：新增内联辅助 `_archive_base_already_has_branch(base, branch)`，判断 archive_base 的直接父目录是否已是当前 branch 且末段为 `archive`；若是 primary 形态且拟拼的 folder 正是当前 branch，则折叠一层，直接用 `archive_base` 作为 `target_parent`；legacy 形态和 folder != current_branch 的场景保持原行为。
- Step 3：历史双层 branch 目录保持不动。修复仅影响"新写入"路径，不触碰既存 `archive/main/main/req-xx/`（AC-05 Excluded 条款）。
- Step 4：新增 `tests/test_archive_path.py`，2 个 tempdir 隔离用例，`mock.patch` 锁定 `_get_git_branch -> "main"` 与 `input -> "n"` 避免 git 依赖与交互。
  - `test_archive_path_no_double_branch`：断言归档落到 `artifacts/main/archive/req-99-demo-archive/`，`archive/main/` 子目录不存在，路径中 `/main/` 段仅出现一次。
  - `test_archive_path_preserves_legacy`：预置历史脏数据 `archive/main/req-88-legacy/MARKER.txt`，执行新归档后断言脏目录与标记文件原封不动，同时新归档落到单层路径。
- 内部测试：`python3 -m unittest tests.test_archive_path -v` → 2/2 OK（0.226s）；全量 `unittest discover tests` → failures=3 errors=3（与 chg-03 基线一致，均为预存问题，零回归）。

## 已知限制
- 修复依赖"archive_base 父目录名 == 当前 branch 且末段为 `archive`"的结构判定。如未来 `resolve_archive_root` 新增第三种形态（例如 `artifacts/{branch}/nested/archive`），需要扩展 `_archive_base_already_has_branch` 的判定规则，否则会再次出现双层拼接。
- 若调用方显式传入 `folder != current_branch`（自定义分组归档），仍会在 archive_base 下拼出 `{archive_base}/{folder}/req-xx`。这是预期行为（保留原 folder 自由度），但需在文档中澄清"folder == branch 时自动折叠"的语义。
