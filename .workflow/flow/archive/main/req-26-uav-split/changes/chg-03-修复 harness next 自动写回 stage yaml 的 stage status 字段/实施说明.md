# 实施说明：chg-03 修复 harness next 自动写回 stage yaml 的 stage / status

## 实际做了什么
- Step 1：定位 `workflow_next` / `workflow_fast_forward` 位于 `src/harness_workflow/workflow_helpers.py`；发现旧实现仅在 runtime.yaml 有 `operation_target` 时才写回 state yaml，缺失时完全跳过——bug 活证的根因。
- Step 2：新增 helper `_sync_stage_to_state_yaml()`（位于 `_get_req_id` 下方），集中处理 requirement / bugfix state yaml 的 stage 写回、`done` 时同步 `status` + `completed_at`、存在 `stage_timestamps` 时打时间戳。风格对齐 `rename_bugfix` / `_state_yaml_for_requirement`。
- Step 2/3：重构 `workflow_next`（4507 附近）与 `workflow_fast_forward`（4568 附近）调用该 helper；关键增强：target 回退顺序为 `operation_target → current_requirement`，解决运行时只有 `current_requirement` 的场景（即现场 req-26 yaml 停留在 `requirement_review` 的 bug）。
- Step 5：新增 `tests/test_next_writeback.py`，3 个用例全部 tempdir 隔离（`test_next_writes_stage_to_requirement_yaml` / `test_next_writes_done_status` / `test_next_writes_bugfix_yaml`）。`python3 -m unittest tests.test_next_writeback -v` 全过。
- 内部回归：全套 `python3 -m unittest discover tests` 从基线 `failures=12 errors=4` 降到 `failures=3 errors=3`（均为预存问题，与本 chg 无关）。

## 未做与原因
- Step 4（regression 状态同步）：未实施。briefing 硬约束"不动 chg-01/02/04 已完成范围"，regression 状态由 chg-01 的 `_update_regression_meta_status` 负责，本 chg 不重叠。
- Step 6（端到端 `test_full_lifecycle.py`）：未实施。briefing 明确"不跑真 CLI，tempdir 隔离"，端到端 smoke 归入 chg-06。
- 失败回滚（plan.md Step 2 第 3 点）：未实施原子回滚。`save_simple_yaml` 目前无事务语义，引入 rollback 需修改 IO 层，超出 chg-03 边界，已在 helper docstring 标注"best-effort"语义。

## 关键文件变更
- `src/harness_workflow/workflow_helpers.py`：新增 `_sync_stage_to_state_yaml()`（L3995-4062 附近）；`workflow_next` 内联 state yaml 双写段（原 L4507-4523）替换为一句 helper 调用；`workflow_fast_forward` 同理。
- `tests/test_next_writeback.py`：新增，3 个 unit test，tempdir 隔离 `.workflow/state/{requirements,bugfixes}/` 结构。

## 已知限制
- 当 `.workflow/state/requirements/{id}.yaml` 文件不存在时，helper 返回 `None` 且不报错（best-effort），与旧行为一致；若需要严格断言，应由调用方在上层加检查。
- state yaml 首次出现 `stage_timestamps` 字段时不主动创建键，避免 schema 漂移；已有字段才会打时间戳。
