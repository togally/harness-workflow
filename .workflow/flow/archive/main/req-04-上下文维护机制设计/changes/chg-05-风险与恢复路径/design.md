# 风险与恢复路径设计

## 1. 概述

本文档定义 harness-workflow 系统的上下文爆炸风险条目和恢复路径，为上下文维护提供完整的风险管理框架。

**设计目标**：
- 在 constraints 层注册上下文爆炸风险
- 定义上下文超限后的标准化恢复路径
- 更新行为边界以规范上下文维护行为
- 提供风险识别和恢复操作指南

**适用范围**：所有使用 Claude Code 作为执行平台的 harness-workflow 会话。

## 2. 风险分析

### 2.1 上下文爆炸风险特征

**触发条件**：
- 上下文长度超过 Claude API 限制（102400 tokens）
- 监控指标达到紧急阈值（>95% 最大上下文）
- 消息条数、文件读取次数等辅助指标超限

**实际影响**：
1. **API调用失败**：返回 400/413 错误，"context length exceeded"
2. **工作流中断**：当前任务无法继续，需要重新开始
3. **数据丢失风险**：未保存的决策、部分完成工作、会话历史
4. **用户挫败感**：时间浪费、进度丢失、需要重复工作

**风险等级**（基于 chg-01 阈值）：
- **低风险**：<70% 最大上下文，正常范围
- **中风险**：70-85%，需要关注
- **高风险**：85-95%，必须执行维护
- **紧急风险**：>95%，立即处理

### 2.2 历史教训与最佳实践

**现有机制**：
- handoff 交接协议（chg-03）
- 失败恢复规则（`constraints/recovery.md`）
- 定期检查机制（session-memory 记录）

**最佳实践**：
1. **预防优于恢复**：定期监控，避免达到紧急阈值
2. **分层恢复策略**：根据风险等级选择合适的维护动作
3. **标准化交接**：使用 handoff.md 协议确保信息完整性
4. **经验沉淀**：记录每次恢复操作的经验教训

## 3. 风险条目设计

### 3.1 `constraints/risk.md` 更新

在 `constraints/risk.md` 中新增 **"上下文爆炸风险"** 章节，包含以下内容：

#### 风险条目：上下文爆炸导致工作流中断

**风险描述**：
上下文长度超过 Claude API 限制（102400 tokens），导致 API 调用失败、工作流完全中断。

**影响分析**：
1. **会话丢失**：当前会话无法继续，需重新开始
2. **任务中断**：正在进行的工作被迫停止
3. **数据不一致**：部分完成的工作可能无法恢复
4. **时间浪费**：需要重复已完成的工作

**缓解措施**：
1. **监控预警**：使用 chg-01 设计的监控指标和阈值
2. **维护动作**：根据 chg-02 决策树选择合适的维护动作
3. **恢复路径**：按照本章节定义的恢复路径执行
4. **handoff 交接**：严重超限时使用 chg-03 handoff 协议

**检测方法**：
1. **主动监控**：主 agent 定期估算上下文负载
2. **阈值触发**：监控指标达到预警/强制维护/紧急阈值
3. **API错误识别**：捕捉 "context length exceeded" 错误
4. **性能下降感知**：响应速度明显变慢

**风险等级**：
- 触发预警阈值：低风险（继续监控）
- 触发强制维护阈值：高风险（必须执行维护）
- 触发紧急阈值：紧急风险（立即处理）

**处理规则**：
1. 低风险：继续监控，提醒用户关注
2. 高风险：必须执行维护动作，不允许继续工作
3. 紧急风险：立即执行维护，优先考虑新开 agent

## 4. 恢复路径设计

### 4.1 `constraints/recovery.md` 更新

在 `constraints/recovery.md` 中新增 **"上下文超限恢复路径"** 章节，定义四阶段恢复流程：

#### 第一阶段：检测阶段
**目标**：识别上下文超限情况

**检测方法**：
1. 主动监控：定期估算 token 使用量
2. 阈值触发：监控指标达到预设阈值
3. 被动发现：API 返回超限错误

**输出**：超限确认报告，包含当前负载、阈值差距、建议动作

#### 第二阶段：决策阶段
**目标**：选择合适的恢复动作

**决策依据**：
1. **当前负载水平**（引用 chg-01 阈值）：
   - 预警范围（70-80%）：建议 `/compact`
   - 强制维护范围（85-90%）：必须维护，根据任务状态选择
   - 紧急范围（>95%）：必须新开 agent
2. **任务状态**：
   - 任务刚开始：可考虑 `/clear` 重新开始
   - 任务接近完成：优先 `/compact` 保留上下文
   - 关键决策已记录：可考虑新开 agent + handoff
3. **可用时间与资源**：
   - 时间充裕：可尝试 `/compact`
   - 时间紧迫：直接 `/clear` 或新开 agent

**决策输出**：明确的维护动作指令（`/compact`、`/clear`、新开 agent）

#### 第三阶段：执行阶段
**目标**：执行选定的维护动作

**执行步骤**：

**选项 A：使用 `/compact`**
1. 确认当前会话历史仍有价值
2. 执行 `/compact` 命令
3. 验证压缩效果，重新估算上下文负载
4. 继续当前任务

**选项 B：使用 `/clear`**
1. 确认不需要保留当前会话历史
2. 记录关键信息到 session-memory
3. 执行 `/clear` 命令
4. 重新加载必要信息，继续工作

**选项 C：新开 agent + handoff**
1. 创建 handoff.md 交接文件（引用 chg-03 模板）
2. 记录任务状态、关键决策、待处理问题
3. 创建新 agent 或新会话
4. 加载 handoff.md 和相关文件
5. 验证交接完整性，继续工作

#### 第四阶段：验证阶段
**目标**：验证恢复成功，工作流可继续

**验证内容**：
1. **上下文负载验证**：重新估算 token 使用量，确认在安全范围内
2. **任务连续性验证**：确认任务状态正确恢复，无数据丢失
3. **功能完整性验证**：系统功能正常，可继续执行任务
4. **性能验证**：响应速度恢复正常水平

**验证输出**：恢复成功报告或失败处理建议

### 4.2 分层恢复策略

基于风险等级设计分层恢复策略：

| 风险等级 | 恢复策略 | 预期恢复时间 | 成功率 |
|----------|----------|--------------|--------|
| **轻度超限**（70-85%） | `/compact` 压缩 | 1-2 分钟 | 高 |
| **中度超限**（85-95%） | 根据任务状态选择：<br>- 任务刚开始 → `/clear`<br>- 任务接近完成 → `/compact`<br>- 关键决策已记录 → handoff | 2-5 分钟 | 中 |
| **严重超限**（>95%） | 必须新开 agent + handoff | 5-10 分钟 | 高 |

## 5. 行为边界更新

### 5.1 `constraints/boundaries.md` 更新

在 `constraints/boundaries.md` 中新增 **"上下文维护行为边界"** 章节，定义四类职责边界：

#### 5.1.1 主 agent 职责边界
**必须执行**：
1. 定期监控上下文负载（阶段转换、任务完成时）
2. 触发维护动作当阈值达到时
3. 记录监控数据和维护历史到 session-memory
4. 协调 subagent 的上下文消耗报告

**禁止行为**：
1. 忽略上下文超限警告继续工作
2. 未经用户确认执行高风险维护动作
3. 不记录维护操作和经验教训

#### 5.1.2 subagent 职责边界
**必须执行**：
1. 报告任务期间的上下文消耗（新增消息、文件读取等）
2. 建议清理时机当上下文负载较高时
3. 遵循主 agent 的维护指令

**禁止行为**：
1. 过度消耗上下文而不报告
2. 自行执行 `/clear` 等高风险命令
3. 不配合交接流程

#### 5.1.3 用户职责边界
**建议行为**：
1. 关注上下文状态提醒
2. 及时响应维护建议
3. 确认高风险维护动作
4. 提供交接所需信息

**可选行为**：
1. 手动检查上下文状态
2. 主动触发维护动作
3. 提供恢复反馈

#### 5.1.4 工具使用边界
**`/compact` 使用边界**：
- 允许：历史消息仍有价值，需要继续当前任务
- 禁止：历史消息已无效，或已尝试压缩但效果不佳

**`/clear` 使用边界**：
- 允许：历史消息已无效或干扰，任务可重新开始
- 禁止：有关键未保存信息，或任务接近完成

**新开 agent 使用边界**：
- 允许：上下文已爆满，无法继续，需要完整交接
- 禁止：轻微超限，可通过简单维护解决

## 7. 使用指南

### 7.1 风险识别指南

**早期迹象识别**：
1. **响应变慢**：AI 回复速度明显下降
2. **记忆模糊**：AI 开始遗忘之前的对话内容
3. **工具调用延迟**：Read/Grep 等工具响应变慢
4. **重复解释**：需要重复之前已说明的信息

**风险等级评估**：
1. **检查监控指标**：token 估算、消息条数、文件读取次数
2. **对比阈值**：判断当前处于哪个风险等级
3. **评估任务状态**：任务进度、关键决策记录情况
4. **考虑时间因素**：可用时间和紧急程度

**触发时机判断**：
- **立即触发**：达到紧急阈值（>95%），或 API 已报错
- **尽快触发**：达到强制维护阈值（85-95%）
- **建议触发**：达到预警阈值（70-85%）且任务仍将继续较长时间

### 7.2 恢复操作指南

**逐步操作说明**：

**步骤 1：确认超限情况**
1. 检查当前上下文负载（token 估算百分比）
2. 确认达到的阈值等级
3. 评估任务状态和进度

**步骤 2：选择恢复动作**
1. 参考 chg-02 决策树
2. 考虑任务状态、可用时间等因素
3. 确定具体维护动作

**步骤 3：执行维护动作**
1. **如果选择 `/compact`**：
   - 执行命令：`/compact`
   - 等待压缩完成
   - 验证压缩效果
2. **如果选择 `/clear`**：
   - 保存关键信息到 session-memory
   - 执行命令：`/clear`
   - 重新加载必要文件
3. **如果选择新开 agent**：
   - 创建 handoff.md 交接文件
   - 记录所有必要信息
   - 创建新 agent/会话
   - 加载交接文件

**步骤 4：验证恢复结果**
1. 重新检查上下文负载
2. 验证任务连续性
3. 测试系统功能
4. 记录恢复经验

**常见问题解决**：

**问题 1：`/compact` 后负载仍高**
- **可能原因**：历史消息过多，压缩空间有限
- **解决方案**：考虑 `/clear` 或新开 agent

**问题 2：`/clear` 后关键信息丢失**
- **预防措施**：执行前务必保存关键信息到 session-memory
- **恢复措施**：从 session-memory 重新加载

**问题 3：handoff 交接失败**
- **可能原因**：交接文件不完整或格式错误
- **解决方案**：检查 handoff.md 模板要求，重新创建

**成功恢复验证标准**：
1. ✅ 上下文负载降至安全范围（<70%）
2. ✅ 任务可继续执行，无数据丢失
3. ✅ 系统功能正常，工具调用无错误
4. ✅ 响应速度恢复正常水平
5. ✅ 恢复经验已记录到 session-memory
