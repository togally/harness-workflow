# 工具层集成设计

## 1. 概述

本文档定义 harness-workflow 系统中 Claude Code 上下文管理工具的集成设计，包括工具 catalog 定义、各阶段工具白名单、选择指南和经验沉淀模板。

**设计目标**：
- 在 tools 层集成 Claude Code 上下文管理能力
- 定义 `/compact`、`/clear`、`/new` 等命令在工作流中的规范用法
- 为各阶段提供明确的上下文维护工具使用指南
- 建立经验沉淀机制，持续优化维护实践

**设计范围**：
- 工具 catalog 条目：`tools/catalog/claude-code-context.md`
- 阶段工具白名单：`tools/stage-tools.md` 更新
- 工具选择指南：`tools/selection-guide.md` 补充
- 经验沉淀模板：`context/experience/tool/claude-code-context.md`

**与相关变更的集成**：
- **chg-01**：引用监控指标和阈值定义
- **chg-02**：引用维护动作决策逻辑
- **chg-03**：引用 handoff 交接协议
- **chg-05**：为风险与恢复路径提供工具支持
- **chg-06**：为经验沉淀与角色职责更新提供工具层基础

## 2. 工具 catalog 设计

### 2.1 工具定义

**工具名称**：Claude Code 上下文管理
**类型**：内置工具
**状态**：active

**核心能力**：
1. **`/compact`**：压缩历史消息，保留重要信息但减少细节
2. **`/clear`**：清除历史消息，重新开始
3. **`/new`**：创建新会话（需配合 handoff 交接）
4. **`/help`**：获取帮助信息

### 2.2 适用场景设计

基于 chg-01 的阈值定义和 chg-02 的决策逻辑，设计以下适用场景：

#### 2.2.1 基于阈值的适用场景
- **预警阈值**（70-80%）：建议使用 `/compact`
- **强制维护阈值**（85-90%）：必须使用维护动作
- **紧急阈值**（>95%）：必须新开 agent

#### 2.2.2 基于任务状态的适用场景
- **任务刚开始**：可考虑 `/clear`
- **任务进行中**：优先 `/compact`
- **任务接近完成**：强烈倾向 `/compact`
- **任务已完成**：可考虑 `/clear`

#### 2.2.3 基于阶段的适用场景
- **planning/executing**：主要使用 `/compact`
- **testing**：根据测试需求选择 `/compact` 或 `/clear`
- **acceptance**：通常使用 `/compact`
- **regression**：复杂问题优先新开 agent

### 2.3 命令详细说明

#### 2.3.1 `/compact` 命令
- **功能**：压缩历史消息，保留重要信息
- **适用场景**：预警阈值、强制维护阈值+任务进行中、强制维护阈值+任务接近完成
- **操作步骤**：执行命令 → 重新读取必要文件 → 验证关键信息保留
- **注意事项**：压缩可能丢失细节，需验证关键信息

#### 2.3.2 `/clear` 命令
- **功能**：清除历史消息，重新开始
- **适用场景**：强制维护阈值+任务刚开始、强制维护阈值+任务已完成
- **操作步骤**：保存重要信息 → 执行命令 → 重新加载所有必要文件
- **风险提示**：所有历史信息丢失，需谨慎使用

#### 2.3.3 `/new` 命令
- **功能**：创建新会话
- **适用场景**：紧急阈值、系统性能问题、用户请求
- **交接要求**：必须创建完整的 handoff 交接文件（遵循 chg-03 协议）
- **特殊要求**：确保所有重要信息已传递

### 2.4 最佳实践设计

#### 2.4.1 基于阈值的维护决策
- 引用 chg-01 阈值定义
- 提供明确的维护动作选择指南
- 考虑多指标组合情况

#### 2.4.2 安全第一原则
- 执行前检查关键信息是否已保存
- 执行后验证维护效果
- 记录维护经验和教训

#### 2.4.3 阶段适配策略
- 不同阶段有不同的维护策略
- 考虑阶段特点和需求
- 提供阶段特定的建议

## 3. 阶段工具白名单设计

### 3.1 各阶段上下文管理工具配置

#### 3.1.1 requirement_review / planning 阶段
- **允许**：`/compact`（当上下文负载达到预警阈值时）
- **限制**：避免使用 `/clear`，除非任务刚开始且上下文混乱
- **注意**：确保关键决策已记录到文件再执行维护

#### 3.1.2 executing 阶段
- **允许**：`/compact`（主要维护动作）、`/clear`（仅任务刚开始时）
- **触发条件**：根据 chg-01 阈值和 chg-02 决策树
- **最佳实践**：定期检查上下文负载，达到阈值时及时维护

#### 3.1.3 testing 阶段
- **允许**：`/compact`、`/clear`
- **适用场景**：阶段转换时建议 `/compact`，测试环境混乱时可考虑 `/clear`
- **限制**：确保测试状态和结果已记录到文件

#### 3.1.4 acceptance 阶段
- **允许**：`/compact`
- **限制**：避免使用 `/clear`，除非验收完全失败需要重新开始
- **注意**：验收阶段通常上下文负载较低，维护需求较少

#### 3.1.5 regression 阶段
- **允许**：新开 agent（主要）、`/compact`、`/clear`
- **适用场景**：复杂问题诊断必须新开 agent，历史分析使用 `/compact`
- **特殊要求**：新开 agent 时必须创建完整的 handoff 交接文件

### 3.2 通用规则设计

#### 3.2.1 触发条件规则
- 直接引用 chg-01 阈值定义
- 提供统一的触发条件判断标准
- 考虑多指标组合情况

#### 3.2.2 决策逻辑规则
- 引用 chg-02 决策逻辑
- 考虑任务状态和阶段因素
- 提供简化的决策支持

#### 3.2.3 执行要求规则
- 标准化的执行前检查清单
- 标准化的重新加载文件清单
- 标准化的效果验证流程

#### 3.2.4 风险控制规则
- 明确各维护动作的风险
- 提供风险应对措施
- 建立风险记录机制

## 8. 验收标准

### 8.1 工具 catalog 验收标准

1. **完整性**：包含所有必要的命令说明、适用场景、最佳实践
2. **准确性**：命令功能描述准确，适用场景合理
3. **实用性**：提供实际可用的操作指南和示例
4. **集成性**：正确引用 chg-01 阈值和 chg-02 决策逻辑

### 8.2 阶段工具白名单验收标准

1. **合理性**：各阶段的工具配置符合阶段特点
2. **明确性**：允许和限制的工具定义明确
3. **实用性**：提供实际可用的阶段适配建议
4. **完整性**：覆盖所有阶段和常见场景

### 8.3 工具选择指南验收标准

1. **全面性**：覆盖所有选择因素和决策场景
2. **实用性**：提供实际可用的决策支持
3. **清晰性**：决策流程图和选择矩阵清晰易懂
4. **一致性**：与 chg-02 决策逻辑保持一致

### 8.4 经验沉淀模板验收标准

1. **完整性**：包含完整的经验记录要素
2. **实用性**：模板易于使用，记录内容有价值
3. **系统性**：提供系统化的经验管理方法
4. **可扩展性**：支持经验积累和持续改进

### 8.5 集成验证验收标准

1. **一致性**：与相关变更的集成关系正确
2. **完整性**：所有必要的集成点已验证
3. **正确性**：引用和集成的逻辑正确
4. **可维护性**：集成关系清晰，易于维护
