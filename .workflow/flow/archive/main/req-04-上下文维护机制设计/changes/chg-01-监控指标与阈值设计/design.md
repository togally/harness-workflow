# 上下文维护监控指标与阈值设计

## 1. 概述

本文档定义 harness-workflow 系统的上下文维护监控指标、触发阈值和采集方法，为后续维护动作提供决策依据。

**设计目标**：
- 量化评估上下文负载状态
- 提供明确的维护触发条件
- 设计可行的指标采集机制
- 为 chg-02（维护动作决策树）提供输入

**适用范围**：所有使用 Claude Code 作为执行平台的 harness-workflow 会话。

## 2. 上下文限制调研结果

### 2.1 Claude API 限制
- **最大上下文长度**：102400 tokens
- **模型限制**：Claude 3.5 Sonnet (20241022) 等版本

### 2.2 Claude Code 内置命令
- **`/compact`**：压缩历史消息，保留重要信息但减少细节
  - 适用场景：历史消息仍有价值但占用空间过多
  - 效果：减少上下文长度，保留关键信息
- **`/clear`**：清除历史消息，重新开始
  - 适用场景：历史消息已无效或干扰新任务
  - 效果：重置上下文，重新加载必要信息

### 2.3 关键限制总结
1. **硬限制**：102400 tokens 为绝对上限，超过会导致 API 错误
2. **软限制**：实际使用中应保留安全边际，避免接近上限
3. **性能影响**：上下文越长，处理速度越慢，成本越高

## 3. 监控指标清单

### 3.1 Token 估算
**定义**：基于消息历史、文件内容等估算当前上下文占用的 token 数量。

**量化方法**：
1. **消息历史估算**：
   - 用户消息：平均 50 tokens/字符（中英文混合）
   - AI 回复：平均 30 tokens/字符
   - 系统提示：固定估算值
2. **文件内容估算**：
   - 代码文件：1.3 tokens/字符
   - 文本文件：1.0 tokens/字符
   - 配置文件：1.0 tokens/字符
3. **工具调用估算**：
   - Read 工具输出：按实际字符数计算
   - Grep 工具输出：按实际字符数计算
   - Bash 工具输出：按实际字符数计算

**采集时机**：
- 阶段转换时（如 planning → executing）
- 任务完成时（subagent 任务结束）
- 用户手动触发检查时
- 定期检查（如每 10 次工具调用后）

### 3.2 消息条数
**定义**：用户消息 + AI 回复的总条数。

**量化方法**：
- 用户消息：每次用户输入计为 1 条
- AI 回复：每次 AI 输出计为 1 条
- 系统消息：不计入统计

**采集时机**：
- 实时统计，每次消息交互后更新
- 阶段转换时记录快照
- 任务完成时汇总报告

### 3.3 文件读取次数
**定义**：Read 工具调用次数。

**量化方法**：
- 每次 Read 工具调用计为 1 次
- 按文件类型分类统计：
  - 代码文件读取次数
  - 文档文件读取次数
  - 配置文件读取次数

**采集时机**：
- 实时统计，每次 Read 调用后更新
- 阶段转换时汇总
- 任务完成时报告

### 3.4 会话时长
**定义**：从会话开始到当前的时间。

**量化方法**：
- 绝对时长：小时:分钟格式
- 相对时长：相对于典型任务时长的百分比

**采集时机**：
- 持续计时，定期更新
- 阶段转换时记录
- 任务完成时汇总

### 3.5 工具调用分布
**定义**：各类工具调用次数的分布情况。

**量化方法**：
- Read：文件读取次数
- Grep：内容搜索次数
- Edit：文件编辑次数
- Bash：命令执行次数
- 其他工具：对应调用次数

**采集时机**：
- 实时统计，每次工具调用后更新
- 阶段转换时汇总分布
- 任务完成时分析报告

## 4. 触发阈值设计

### 4.1 阈值层级

基于最大上下文长度（102400 tokens）设计三层阈值：

#### 4.1.1 预警阈值（黄色警报）
- **范围**：70-80% 最大上下文（71680-81920 tokens）
- **含义**：上下文负载较高，需要关注
- **建议动作**：提醒用户上下文负载情况，建议考虑维护
- **触发条件**：任一监控指标达到预警范围

#### 4.1.2 强制维护阈值（橙色警报）
- **范围**：85-90% 最大上下文（87040-92160 tokens）
- **含义**：上下文负载接近危险水平，必须执行维护
- **建议动作**：必须执行维护动作（`/compact` 或 `/clear`）
- **触发条件**：Token 估算达到强制维护范围

#### 4.1.3 紧急阈值（红色警报）
- **范围**：>95% 最大上下文（>97280 tokens）
- **含义**：上下文已接近爆炸边缘，需立即处理
- **建议动作**：立即执行维护动作，优先考虑 `/clear` 或新开 agent
- **触发条件**：Token 估算达到紧急范围

### 4.2 辅助阈值

#### 4.2.1 消息条数阈值
- **预警**：>100 条消息
- **强制维护**：>150 条消息
- **紧急**：>200 条消息

#### 4.2.2 文件读取次数阈值
- **预警**：>50 次读取
- **强制维护**：>80 次读取
- **紧急**：>100 次读取

#### 4.2.3 会话时长阈值
- **预警**：>2 小时
- **强制维护**：>4 小时
- **紧急**：>6 小时

## 5. 指标采集方法

### 5.1 主 agent 定期估算
**采集方式**：主 agent 在关键节点主动估算上下文负载。

**触发条件**：
- 阶段转换时（requirement_review → planning 等）
- 任务完成时（subagent 返回控制权）
- 用户执行特定命令时
- 定期时间间隔（如每 30 分钟）

**采集内容**：
- Token 估算值
- 消息条数统计
- 文件读取次数
- 会话时长
- 工具调用分布

### 5.2 subagent 任务结束报告
**采集方式**：每个 subagent 在任务结束时报告消耗的上下文量。

**报告时机**：
- 任务成功完成时
- 任务失败退出时
- 任务被中断时

**报告内容**：
- 任务期间新增消息条数
- 任务期间文件读取次数
- 任务期间工具调用分布
- 任务耗时
- 估算的任务上下文增量

### 5.3 用户手动检查点
**采集方式**：用户可通过命令随时检查当前上下文负载。

**检查命令**：`harness context-status`（待实现）

**输出内容**：
- 当前上下文负载状态（百分比）
- 各监控指标当前值
- 距离各阈值的差距
- 建议的维护动作

### 5.4 采集频率优化
**高频采集**（实时）：
- 消息条数：每次消息交互后
- 工具调用：每次工具调用后

**中频采集**（关键节点）：
- Token 估算：阶段转换、任务完成时
- 文件读取：阶段转换时汇总

**低频采集**（定期）：
- 完整状态报告：每 30 分钟或用户请求时

## 6. 验证方法

### 6.1 验证场景设计

#### 场景 1：正常负载下的指标计算
- **模拟条件**：50 条消息，30 次文件读取，1 小时会话
- **验证目标**：指标计算准确，阈值判断正确
- **预期结果**：显示正常负载，无警报

#### 场景 2：预警阈值触发
- **模拟条件**：120 条消息，60 次文件读取，2.5 小时会话
- **验证目标**：预警阈值正确触发
- **预期结果**：显示黄色警报，建议关注

#### 场景 3：强制维护阈值触发
- **模拟条件**：180 条消息，90 次文件读取，4.5 小时会话
- **验证目标**：强制维护阈值正确触发
- **预期结果**：显示橙色警报，要求执行维护

#### 场景 4：紧急阈值触发
- **模拟条件**：220 条消息，110 次文件读取，6.5 小时会话
- **验证目标**：紧急阈值正确触发
- **预期结果**：显示红色警报，要求立即处理

### 6.2 验收标准

1. **完整性**：design.md 包含所有要求的指标、阈值和采集方法
2. **清晰性**：指标定义清晰，量化方法明确
3. **合理性**：阈值设计符合实际使用场景，有安全边际
4. **可行性**：采集方法考虑实现成本和性能影响
5. **一致性**：与后续变更（chg-02、chg-05）的接口明确

### 6.3 验证步骤

1. 人工评审 design.md 的完整性和合理性
2. 模拟不同负载场景下的指标计算
3. 测试阈值触发逻辑的正确性
4. 验证采集方法的可行性
5. 检查与后续变更的接口一致性

## 7. 与后续变更的接口

### 7.1 chg-02 维护动作决策树
- **输入**：本设计定义的监控指标和阈值
- **输出**：根据阈值触发相应的维护动作（`/compact`、`/clear`、新开 agent）
- **接口**：阈值判断 → 动作决策的映射关系

### 7.2 chg-04 工具层集成
- **输入**：指标采集需要的工具支持
- **输出**：Claude Code 上下文管理工具的集成规范
- **接口**：采集方法 → 工具调用的实现方式

### 7.3 chg-05 风险与恢复路径
- **输入**：阈值触发的风险场景
- **输出**：上下文爆炸的恢复路径
- **接口**：紧急阈值 → 恢复动作的触发条件

### 7.4 chg-06 经验沉淀与角色职责更新
- **输入**：监控指标的实际使用经验
- **输出**：角色职责中关于上下文监控的部分
- **接口**：采集方法 → 角色职责的具体要求

## 8. 附录

### 8.1 Token 估算参考值
- 英文文本：1 token ≈ 4 字符
- 中文文本：1 token ≈ 2 字符
- 代码文件：1 token ≈ 3-4 字符（因符号和格式）
- 混合内容：按实际比例加权计算

### 8.2 安全边际建议
- 预警阈值：保留 20-30% 安全空间
- 强制维护阈值：保留 10-15% 安全空间
- 紧急阈值：保留 5% 绝对最小空间

### 8.3 性能考虑
- 实时采集：轻量级统计，避免影响性能
- 定期估算：可安排在空闲时段
- 用户检查：按需执行，不增加额外负担
