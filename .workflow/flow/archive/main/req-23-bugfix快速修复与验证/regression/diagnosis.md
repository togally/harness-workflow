# Regression Diagnosis: req-23

## 问题描述

用户在 planning 阶段触发 regression，提出：
> "在本次需求中插入一个小功能点，我想要在角色的基础约束中加一条，每次角色执行前告诉用户他是谁、他准备做什么。"

## 证据

- 当前 `base-role.md` 中已存在"硬门禁二：操作说明与日志"，要求执行操作前说明操作名称和结果
- 当前 `base-role.md` 中没有要求角色在开始执行任务前向用户做自我介绍（身份 + 任务意图）
- 这不是 req-23 验收标准中已定义的缺陷或未满足项，而是用户在 planning 阶段主动提出的范围扩展

## 根因分析

1. **不是缺陷**：req-23 的需求文档（requirement.md）中并未包含"角色自我介绍"这一约束要求，当前系统也不存在与此相关的实现错误。
2. **需求范围扩展**：用户在拆分变更进入 planning 后，意识到增加一条基础角色约束可以提升交互透明度，因此主动追加需求范围。
3. **边界判断**：这属于 req-23 范围内的合理增强（修改 `base-role.md` 即可），不需要拆分为独立需求；也不属于实现/测试问题，不需要回退到 testing。

## 结论

- [ ] 真实问题
- [x] 误判（不是缺陷，是需求范围扩展请求）

## 路由决定

- **问题类型**：误判 / 需求范围扩展
- **目标阶段**：planning（回到触发前的 stage）

## 修复方向/处理建议

1. 更新 req-23 的 `requirement.md`，在"范围-包含"中补充：
   > 更新 `base-role.md`，增加角色自我介绍规则：每次角色执行前须向用户说明自身身份和当前任务意图。

2. 将该约束纳入 chg-01（定义 bugfix 目录规范与模板）或独立为 chg-04（base-role 约束增强），视变更拆分结果而定。

3. 在 `base-role.md` 的"通用准则"或 SOP 章节中新增：
   > 每次角色开始执行实质性任务前，须向用户简要说明："我是 [角色名称]，接下来我将 [任务意图]。"
