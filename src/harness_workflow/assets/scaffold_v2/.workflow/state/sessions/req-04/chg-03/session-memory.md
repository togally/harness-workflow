## 经验沉淀

### 经验 1：模板设计应基于实际使用场景
- **场景**：设计 handoff 协议模板时，需要同时考虑简单场景和复杂场景的需求
- **经验内容**：
  - 采用必填字段和可选字段分离的设计，简单场景只需填写必填字段，复杂场景可充分利用可选字段
  - 提供完整版和简化版示例，展示不同场景下的模板使用
  - 设计时考虑实际填写成本，避免过度设计导致模板难以使用
- **可泛化性**：适用于所有需要设计模板或表单的场景，平衡完整性和易用性

### 经验 2：系统集成设计应明确接口定义
- **场景**：handoff 协议需要与 chg-02 决策树和 chg-05 恢复路径集成
- **经验内容**：
  - 明确接口定义：触发条件、输入、输出、流程
  - 设计验证方法：如何验证集成的正确性和完整性
  - 提供使用指南：具体的使用步骤和最佳实践
  - 考虑错误处理：集成失败时的恢复路径
- **可泛化性**：适用于所有需要与系统其他组件集成的变更设计

### 经验 3：文档结构应支持渐进式理解
- **场景**：design.md 文档需要同时满足快速查阅和深入理解的需求
- **经验内容**：
  - 采用分层结构：概述 → 详细设计 → 示例 → 验证 → 附录
  - 提供速查表：关键信息的快速参考
  - 包含常见问题解答：解决典型疑问
  - 使用清晰的标题和标记：便于导航和理解
- **可泛化性**：适用于所有技术设计文档的编写，提高文档的可用性

### 经验 4：变更执行应严格遵循计划步骤
- **场景**：执行 plan.md 中的 6 个步骤，确保不遗漏任何关键任务
- **经验内容**：
  - 按步骤顺序执行，每步完成后在 session-memory 中标记
  - 每步产出明确，便于验证和追踪
  - 步骤间有逻辑依赖，前一步为后一步提供基础
  - 完成所有步骤后进行综合测试，确保整体质量
- **可泛化性**：适用于所有需要按计划执行的变更开发工作
