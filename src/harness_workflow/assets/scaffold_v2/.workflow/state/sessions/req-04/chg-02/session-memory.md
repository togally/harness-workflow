## 执行日志

### 步骤 1：分析维护动作特性
- [x] 1.1 研究 `/compact` 命令
  - 工作原理：压缩历史消息，保留重要信息但减少细节
  - 适用场景：历史消息仍有价值但占用空间过多，任务尚未完成需要保留部分上下文
  - 限制：压缩后可能丢失一些细节信息，但保留关键决策和状态
- [x] 1.2 研究 `/clear` 命令  
  - 工作原理：清除历史消息，重新开始
  - 适用场景：历史消息已无效或干扰新任务，需要全新上下文重新开始
  - 风险：清空后无法恢复历史信息，需要重新加载必要文件
- [x] 1.3 研究新开 agent 流程
  - 如何创建新 agent：通过 Claude Code 的新会话功能
  - 交接要求：需要传递任务状态、关键决策、待处理问题、必须传递的文件
  - 与 handoff 协议（chg-03）的集成：使用 handoff.md 协议模板进行标准化交接

### 步骤 2：设计决策树输入输出
- [x] 2.1 定义决策树输入
  - 当前上下文负载（token 估算百分比）：基于 chg-01 的估算方法
  - 消息条数：用户消息 + AI 回复的总条数
  - 文件读取次数：Read 工具调用次数
  - 会话时长：从会话开始到当前的时间
  - 当前任务状态：阶段（stage）、进度、关键决策是否已记录、是否有未保存的重要信息
- [x] 2.2 定义决策树输出
  - 建议的维护动作：`/compact`、`/clear`、新开 agent
  - 动作优先级：建议（可延迟）、必须（立即执行）、紧急（立即处理）
  - 预期效果说明：压缩后保留的信息、清空后需要重新加载的内容、交接需要传递的信息

### 步骤 3：创建决策逻辑
- [x] 3.1 基于 chg-01 阈值设计决策规则
  - **预警阈值范围**（70-80%）：建议使用 `/compact`，继续当前任务
  - **强制维护阈值**（85-90%）：必须执行维护动作，根据任务状态选择 `/compact` 或 `/clear`
  - **紧急阈值**（>95%）：必须新开 agent，使用 handoff 协议交接
- [x] 3.2 考虑任务状态因素
  - 任务刚开始 vs 任务接近完成：刚开始时更倾向于 `/clear`，接近完成时更倾向于 `/compact`
  - 关键决策是否已记录：已记录的关键决策可考虑 `/compact`，未记录则优先 `/clear` 或新开 agent
  - 是否有未保存的重要信息：有未保存信息时优先 `/compact` 或新开 agent 交接
- [x] 3.3 设计决策流程图或决策表
  - 设计决策表（将在 design.md 中详细呈现）
  - 决策流程：检查阈值 → 评估任务状态 → 选择维护动作 → 提供操作指引

### 步骤 4：定义各动作适用场景
- [x] 4.1 `/compact` 适用场景详细定义
  - 历史消息较多但仍有压缩空间：消息条数 >100 但 <200
  - 任务尚未完成，需要保留部分上下文：当前阶段为 executing/testing，关键决策已记录
  - 操作步骤：执行 `/compact` 命令，重新读取必要文件（runtime.yaml、session-memory 等）
  - 注意事项：压缩后需验证关键信息是否保留，必要时补充读取
- [x] 4.2 `/clear` 适用场景详细定义
  - 历史消息已无效或干扰：消息历史与当前任务无关
  - 需要全新上下文重新开始：任务切换、阶段转换、上下文混乱
  - 操作步骤：执行 `/clear` 命令，重新加载所有必要文件
  - 风险提示：所有历史信息丢失，需确保关键信息已保存到文件
- [x] 4.3 新开 agent 适用场景详细定义
  - 上下文已爆满，无法继续：token 估算 >95% 或消息条数 >200
  - 触发条件：紧急阈值触发，且任务必须继续
  - 交接要求：创建 handoff.md，传递任务状态、关键决策、待处理问题、必须文件
  - handoff 协议引用：遵循 chg-03 的 handoff.md 模板
