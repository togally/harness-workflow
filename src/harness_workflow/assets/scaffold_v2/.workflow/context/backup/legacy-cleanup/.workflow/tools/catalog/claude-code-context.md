# 工具：Claude Code 上下文管理

**类型：** 内置工具
**状态：** active

## 适用场景

1. **上下文负载过高时**
   - Token 估算达到预警阈值（70-80% 最大上下文）
   - 消息条数超过 100 条
   - 文件读取次数超过 50 次
   - 会话时长超过 2 小时

2. **任务切换或阶段转换时**
   - 从 `executing` 阶段转换到 `testing` 阶段
   - 从 `done` 阶段进入新 `requirement_review`
   - 从 `regression` 切换回正常开发流程

3. **系统性能下降时**
   - 响应速度明显下降
   - 工具调用失败率增加
   - 内存或性能问题

## 不适用场景

1. **关键决策未记录时**
   - 重要技术决策未保存到文件
   - 任务状态未记录到 `session-memory.md`
   - 有未保存的重要信息

2. **简单任务或快速验证**
   - 上下文负载较低（<50% 最大上下文）
   - 任务简单，不需要复杂历史
   - 快速验证某个功能或修复

3. **任务接近完成时（除非紧急）**
   - 任务已完成 80% 以上
   - 只需少量收尾工作
   - 历史上下文对完成任务仍有价值

## 推荐 prompt 模式

| 任务类型 | 调用方式 | 示例 |
|---------|---------|------|
| **压缩历史消息** | `/compact` | `/compact`（无参数） |
| **清空历史重新开始** | `/clear` | `/clear`（无参数） |
| **创建新会话** | `/new` | `/new`（需配合 handoff 交接） |
| **获取帮助信息** | `/help` | `/help context` |

## 命令详细说明

### `/compact` 命令

**功能：** 压缩历史消息，保留重要信息但减少细节

**适用场景（引用 chg-02 决策树）：**
1. **预警阈值触发**：Token 估算 70-80%，消息条数 100-150，任务进行中
2. **强制维护阈值 + 任务进行中**：Token 估算 85-90%，任务进行中，关键决策已记录
3. **强制维护阈值 + 任务接近完成**：Token 估算 85-90%，任务接近完成，关键决策已记录

**操作步骤：**
1. 执行 `/compact` 命令
2. 重新读取必要文件：
   - `.workflow/state/runtime.yaml`
   - 当前 `session-memory.md`
   - 当前变更的 `change.md`、`plan.md`、`design.md`
   - 其他必要配置文件
3. 验证关键信息是否保留
4. 如有遗漏，补充读取相关文件

**注意事项：**
- 压缩后可能丢失一些细节信息
- 需确保关键决策和状态信息已保留
- 如有不确定，可先尝试 `/compact`，观察效果
- 压缩后如发现问题，可考虑进一步维护

### `/clear` 命令

**功能：** 清除历史消息，重新开始

**适用场景（引用 chg-02 决策树）：**
1. **强制维护阈值 + 任务刚开始**：Token 估算 85-90%，任务刚开始
2. **强制维护阈值 + 任务已完成**：Token 估算 85-90%，任务已完成
3. **历史消息已无效或干扰**：上下文混乱，影响新任务执行

**操作步骤：**
1. **确保重要信息已保存**：
   - 关键决策记录到文件
   - 任务状态保存到 `session-memory.md`
   - 必要文件已更新
2. 执行 `/clear` 命令
3. 重新加载所有必要文件：
   - `.workflow/state/runtime.yaml`
   - `.workflow/context/index.md`
   - 当前角色文件（如 `executing.md`）
   - 相关经验文件
   - 当前变更的所有文档
4. 重新建立上下文

**风险提示：**
- **所有历史信息丢失**：无法恢复之前的对话历史
- **需要重新加载文件**：增加初始设置时间
- **可能遗漏信息**：如果未提前保存关键信息
- **适用性有限**：不适用于复杂、进行中的任务

### `/new` 命令

**功能：** 创建新会话

**适用场景（引用 chg-02 决策树）：**
1. **紧急阈值触发**：Token 估算 >95%，消息条数 >200，文件读取次数 >100
2. **系统性能问题**：明显的系统性能下降
3. **用户请求**：用户主动要求新开 agent

**交接要求（引用 chg-03 handoff 协议）：**
必须创建 `handoff.md` 文件，包含以下内容：

1. **任务状态摘要**：
   - 当前 requirement 和 change
   - 当前 stage 和进度
   - 已完成的工作
   - 待完成的工作

2. **关键决策记录**：
   - 重要技术决策
   - 设计选择理由
   - 已解决的问题
   - 未解决的问题

3. **必须传递的文件**：
   - `session-memory.md`
   - 当前变更的 `change.md`、`plan.md`、`design.md`
   - 相关配置文件
   - 生成的代码或文档

4. **接收方指引**：
   - 如何加载交接内容
   - 下一步动作
   - 注意事项和风险提示

### `/help` 命令

**功能：** 获取帮助信息

**适用场景：**
- 了解命令用法
- 查看可用选项
- 获取使用指导

**常用查询：**
- `/help compact`：查看 `/compact` 命令帮助
- `/help clear`：查看 `/clear` 命令帮助
- `/help context`：查看上下文管理相关命令

## 最佳实践

### 1. 基于阈值的维护决策

根据 chg-01 定义的监控指标和阈值，选择适当的维护动作：

| 阈值级别 | 任务状态 | 建议动作 | 说明 |
|----------|----------|----------|------|
| **预警阈值** (70-80%) | 任何状态 | `/compact` | 压缩历史消息，保留关键信息 |
| **强制维护阈值** (85-90%) | 任务刚开始 | `/clear` | 清空历史，重新开始 |
| **强制维护阈值** (85-90%) | 任务进行中 | `/compact` | 压缩历史，保留关键信息 |
| **强制维护阈值** (85-90%) | 任务接近完成 | `/compact` | 压缩历史，完成收尾 |
| **紧急阈值** (>95%) | 任何状态 | **新开 agent** | 创建 handoff 交接 |

### 2. 安全第一原则

1. **执行前检查**：
   - 确认关键决策已记录到文件
   - 验证任务状态已保存到 `session-memory.md`
   - 检查必要文件已更新

2. **执行后验证**：
   - 验证关键信息是否保留
   - 检查上下文负载是否降低
   - 确认任务可继续执行

3. **风险应对**：
   - 如有问题，及时恢复或补救
   - 记录维护经验和教训
   - 优化后续维护决策

### 3. 阶段适配策略

根据当前工作流阶段调整维护策略：

| Stage | 推荐维护动作 | 限制 |
|-------|-------------|------|
| **planning/executing** | `/compact` | 避免 `/clear`，除非任务刚开始 |
| **testing/acceptance** | `/compact` 或 `/clear` | 根据测试需求选择 |
| **regression** | 新开 agent | 复杂问题需要完整交接 |
| **done** | `/clear` | 准备下一个任务 |

### 4. 经验记录要求

每次使用上下文管理工具后，应在 `context/experience/tool/claude-code-context.md` 中记录：

1. **使用场景**：为什么需要维护
2. **选择理由**：为什么选择该维护动作
3. **执行过程**：如何执行的
4. **效果评估**：维护后的效果如何
5. **经验教训**：学到了什么，如何改进

## 与决策树集成

本工具的使用应遵循 chg-02 设计的维护动作决策树：

1. **输入**：chg-01 的监控指标和阈值
2. **决策**：根据阈值和任务状态选择维护动作
3. **执行**：使用本工具执行选定的维护动作
4. **验证**：检查维护效果，记录经验

**决策流程引用：**
```
开始
  ↓
检查上下文负载（Token%、消息数、读取数、时长）
  ↓
判断阈值级别（引用 chg-01）
  ├── 未达阈值 → 继续工作
  ├── 预警阈值 → 建议 `/compact`
  ├── 强制维护阈值 → 必须执行维护
  └── 紧急阈值 → 必须新开 agent
  ↓
评估任务状态（阶段、进度、决策记录）
  ↓
选择具体动作（引用本工具文档）
  ├── `/compact`：压缩保留关键信息
  ├── `/clear`：清空重新开始
  └── 新开 agent：交接继续
  ↓
执行动作并提供指引
  ↓
结束
```

## 注意事项

1. **工具限制**：
   - `/compact` 的压缩算法不可控，可能丢失重要细节
   - `/clear` 会清除所有历史，无法恢复
   - 新会话需要手动创建完整的 handoff 交接

2. **用户责任**：
   - 用户需根据实际情况判断是否需要维护
   - 用户需确保重要信息已保存再执行维护
   - 用户需验证维护后的效果

3. **性能考虑**：
   - 频繁维护可能影响工作效率
   - 延迟维护可能增加风险
   - 需要平衡维护频率和任务连续性

4. **经验积累**：
   - 记录每次维护决策和效果
   - 根据经验优化阈值和决策逻辑
   - 分享最佳实践和教训
