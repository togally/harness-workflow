## 3. 角色职责设计

### 3.1 分工原则

| 角色 | 职责 | 关键词 |
|------|------|--------|
| 主 agent（WORKFLOW.md） | 监控、决策、协调、交接管理 | 主动发起 |
| Subagent（各阶段角色） | 消耗报告、清理建议、状态保存 | 被动上报 |

### 3.2 主 agent 职责扩展（WORKFLOW.md）

在"职责外问题处理"之前新增 `## 上下文维护职责`，包含四项：

- **监控职责**：参考 chg-01 三层阈值（预警70-80%、强制维护85-90%、紧急>95%）及辅助指标（消息数、文件读取次数）；检查时机：阶段转换时、subagent 返回控制权时、定期（每30分钟或每10次工具调用后）
- **触发职责**：按阈值级别主动提醒用户（预警→建议关注，强制维护→必须执行，紧急→立即处理）
- **协调职责**：根据 chg-02 决策树，协调 `/compact`、`/clear`、新开 agent；重要动作执行前需用户确认
- **交接管理**：新开 agent 时按 chg-03 handoff 协议执行四步交接

### 3.3 Subagent 职责扩展（六个角色文件）

在每个角色文件"职责外问题"之前统一新增 `## 上下文维护职责`，核心三点：

1. **消耗报告**：任务完成后报告上下文消耗（文件读取次数、工具调用次数、大文件读取情况）
2. **清理建议**：发现上下文消耗较大时，主动建议主 agent 执行 `/compact`
3. **状态保存**：阶段结束前确认关键决策已保存到 session-memory.md，确保维护后可恢复

各阶段特点微调：
- **executing.md**：额外注明达强制维护阈值时立即上报（执行阶段上下文积累最快）
- **regression.md**：额外注明 regression 本身上下文积累长，需特别关注

## 4. context/index.md 集成

### 4.1 评估结论

原 `context/index.md` 的 before-task 区域（Step 4）仅定义团队规范加载，无上下文监控检查点。

**决策**：更新 Step 4，新增上下文负载检查步骤。

### 4.2 新增内容

在 Step 4（before-task）中增加：

> 在开始实质性任务前，检查当前上下文负载：
> - 消息条数、文件读取次数、会话时长是否触及 chg-01 阈值
> - 如达到预警阈值（70-80%），提醒用户考虑维护
> - 如达到强制维护阈值（≥85%），必须先执行维护再继续任务

同步更新加载顺序速查流程图，在 `team/development-standards.md` 后增加 `上下文负载检查` 节点。
