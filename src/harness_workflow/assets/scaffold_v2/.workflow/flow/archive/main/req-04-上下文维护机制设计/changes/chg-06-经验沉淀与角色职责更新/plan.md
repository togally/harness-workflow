# 执行计划

## 依赖关系
- **前置依赖**：chg-01、chg-02、chg-03、chg-04、chg-05（全部前置变更）
- **后置依赖**：无，这是最后一个变更

## 执行步骤

### 步骤 1：分析本次上下文爆炸教训
1. 收集本次上下文爆炸的完整信息：
   - **现象**：具体发生了什么问题
   - **根因**：为什么会发生上下文爆炸
   - **影响**：对工作流造成了什么影响
   - **解决方案**：如何解决的，效果如何
2. 识别可泛化的教训：
   - 哪些是本次特有的
   - 哪些可能在其他场景重复发生
   - 哪些需要纳入标准流程

### 步骤 2：创建/更新经验沉淀文件
1. 确定经验文件位置：
   - 选项 A：更新 `context/experience/tool/claude-code.md`
   - 选项 B：创建新文件 `context/experience/stage/context-maintenance.md`
   - 基于现有文件结构和内容决定
2. 记录经验内容：
   - **教训总结**：本次上下文爆炸的核心教训
   - **最佳实践**：上下文维护的最佳做法
   - **常见错误**：需要避免的错误做法
   - **工具使用经验**：`/compact`、`/clear` 等命令的实际使用经验
   - **与工作流集成**：如何将上下文维护融入工作流
3. 设计经验记录模板（如需要）

### 步骤 3：更新主 agent 角色职责
1. 分析主 agent 在上下文维护中的职责：
   - 上下文监控：定期检查上下文负载
   - 阈值触发：在达到阈值时提醒用户或自动触发维护
   - 维护动作协调：协调 `/compact`、`/clear` 或新开 agent
   - 交接管理：管理 handoff 协议的执行
2. 更新主 agent 角色文件：
   - 在职责部分增加上下文维护相关职责
   - 在工具使用部分增加上下文管理工具
   - 在职责外问题部分增加上下文相关问题的上报流程

### 步骤 4：更新各阶段 subagent 角色职责
1. 分析各阶段 subagent 的上下文相关职责：
   - **planning/executing**：报告任务消耗的上下文量
   - **testing/acceptance**：在测试前后清理上下文
   - **regression**：处理复杂问题时的上下文管理
2. 更新各阶段角色文件：
   - 增加上下文消耗报告职责
   - 增加清理建议职责
   - 增加与主 agent 的上下文状态沟通职责

### 步骤 5：更新 context/index.md 加载顺序（可选）
1. 分析是否需要增加上下文检查点：
   - 在 Step 1（读取运行时状态）后检查上下文负载
   - 在 before-task 时检查上下文负载
   - 在阶段转换时检查上下文负载
2. 如需要，更新 `context/index.md`：
   - 增加上下文检查点的位置和逻辑
   - 定义检查频率和触发条件
   - 定义超阈值时的处理流程

### 步骤 6：集成验证
1. 验证经验沉淀的完整性和实用性
2. 验证角色职责更新的合理性和一致性
3. 验证与前置变更的引用关系
4. 创建端到端验证场景

## 预期产物
1. 更新/创建的**经验沉淀文件**：`context/experience/tool/claude-code.md` 或 `context/experience/stage/context-maintenance.md`
2. 更新的**主 agent 角色文件**：增加上下文维护职责
3. 更新的**各阶段 subagent 角色文件**：增加上下文相关职责
4. 可选的**更新的 `context/index.md`**：增加上下文检查点
5. `design.md`：集成设计说明

## 验证方法
1. 人工评审经验内容的完整性和准确性
2. 检查角色职责更新的合理性和可执行性
3. 验证与前置变更的集成是否完整
4. 模拟端到端场景验证整体效果

## 注意事项
1. 经验沉淀要具体、可复用，避免空泛描述
2. 角色职责更新要明确、可执行，避免模糊表述
3. 考虑向后兼容性，避免破坏现有工作流
4. 确保各角色职责之间协调一致
5. 参考已有经验文件（如 requirement.md、harness.md）的格式和风格
