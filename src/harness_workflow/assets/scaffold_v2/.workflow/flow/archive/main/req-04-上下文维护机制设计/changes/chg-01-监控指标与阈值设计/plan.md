# 执行计划

## 依赖关系
- 无前置依赖
- 为 chg-02、chg-04、chg-05、chg-06 提供基础指标和阈值

## 执行步骤

### 步骤 1：调研 Claude Code 上下文限制
1. 查阅 Claude Code 文档，确认最大上下文长度（token 数）
2. 了解 `/compact`、`/clear` 等命令的工作原理和效果
3. 记录关键限制：最大 token 数、消息条数限制、文件读取限制等

### 步骤 2：设计监控指标清单
1. 创建 `design.md` 文件，定义以下指标：
   - **Token 估算**：基于消息历史长度、文件内容大小的估算公式
   - **消息条数**：用户消息 + AI 回复的总条数
   - **文件读取次数**：Read 工具调用次数
   - **会话时长**：从会话开始到当前的时间
2. 为每个指标定义量化方法和采集时机

### 步骤 3：设计触发阈值
1. 在 `design.md` 中定义三层阈值：
   - **预警阈值**（如 70-80% 最大上下文）：提醒用户上下文负载较高
   - **强制维护阈值**（如 85-90%）：必须执行维护动作
   - **紧急阈值**（>95%）：上下文已接近爆炸，需立即处理
2. 为每个阈值提供具体数值建议和依据

### 步骤 4：定义指标采集方法
1. 设计三种采集方式：
   - **主 agent 定期估算**：在关键节点（阶段转换、任务完成）估算 token
   - **subagent 任务结束报告**：每个 subagent 任务结束时报告消耗的上下文量
   - **用户手动检查点**：用户可随时通过命令检查当前上下文负载
2. 定义采集频率和触发条件

### 步骤 5：创建验证方法
1. 设计验证场景：
   - 模拟不同上下文负载下的指标计算
   - 测试阈值触发逻辑
   - 验证采集方法的可行性
2. 定义验收标准

## 预期产物
1. `design.md`：包含完整的指标定义、阈值设计、采集方法
2. 验证用例文档（可选）

## 验证方法
1. 人工评审 `design.md` 的完整性和合理性
2. 检查指标定义是否清晰、可量化
3. 验证阈值设计是否符合实际使用场景
4. 确认采集方法是否可行

## 注意事项
1. 保持指标简单实用，避免过度复杂化
2. 阈值设计要考虑实际使用中的安全边际
3. 采集方法要考虑实现成本和性能影响
4. 与后续变更（chg-02、chg-05）的接口要明确
