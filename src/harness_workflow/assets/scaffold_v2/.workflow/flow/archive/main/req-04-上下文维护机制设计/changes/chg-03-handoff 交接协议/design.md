# Handoff 交接协议设计说明

## 1. 概述

### 1.1 设计目标
创建标准化的 `handoff.md` 协议模板，用于在新开 agent 时进行上下文交接，确保任务连续性。

### 1.2 设计原则
1. **完整性**：覆盖任务继续所需的所有关键信息
2. **实用性**：模板易于填写和使用
3. **标准化**：统一格式，便于自动化处理
4. **可扩展性**：支持不同场景的变体需求
5. **集成性**：与工作流其他组件良好集成

### 1.3 适用范围
- 新开 agent 时的上下文交接
- 任务中断后的恢复
- 团队协作时的任务传递
- 长期任务的阶段性存档

## 2. 模板结构设计

### 2.1 文件结构
```
handoff.md
├── 摘要 (必填)
├── 1. 任务状态 (必填)
│   ├── 1.1 当前上下文
│   ├── 1.2 已完成的工作
│   └── 1.3 待完成的工作
├── 2. 关键决策记录 (必填)
│   ├── 2.1 重要技术决策
│   ├── 2.2 已解决的问题
│   └── 2.3 未解决的问题
├── 3. 必须传递的文件 (必填)
│   ├── 3.1 核心状态文件
│   ├── 3.2 变更相关文件
│   ├── 3.3 角色与经验文件
│   └── 3.4 产出文件
├── 4. 接收方指引 (必填)
│   ├── 4.1 加载步骤
│   ├── 4.2 继续工作的步骤
│   └── 4.3 注意事项
├── 5. 会话记忆摘要 (可选)
│   ├── 5.1 关键对话片段
│   ├── 5.2 重要发现
│   └── 5.3 经验教训
├── 6. 元数据 (必填)
└── 7. 联系方式 (可选)
```

### 2.2 必填字段说明

#### 2.2.1 摘要部分
- **交接原因**：必须说明为什么需要交接（上下文爆炸、性能问题、用户请求等）
- **任务概览**：简要描述当前任务，帮助接收方快速理解
- **交接时间**：YYYY/MM/DD HH:MM 格式，用于追踪
- **交接方**：当前 agent 标识（如 Claude Code 会话特征）
- **接收方**：预期接收方标识（如 "新开 Claude Code 会话"）

#### 2.2.2 任务状态
- **当前上下文**：
  - 需求 (Requirement)：`req-XX` 格式
  - 变更 (Change)：`chg-XX-变更标题` 格式
  - 阶段 (Stage)：`executing`、`testing`、`acceptance` 等
  - 进度评估：`刚开始`、`进行中`、`接近完成`、`已完成`
  - 会话模式：`harness` 或 `open`
- **已完成的工作**：使用复选框列表，已完成的标记为 `[x]`
- **待完成的工作**：使用复选框列表，未完成的标记为 `[ ]`

#### 2.2.3 关键决策记录
- **重要技术决策**：每个决策包含标题、理由、影响、相关文件
- **已解决的问题**：问题描述、解决方案、相关文件
- **未解决的问题**：问题描述、当前状态、下一步建议

#### 2.2.4 必须传递的文件
- **核心状态文件**：`runtime.yaml`、`session-memory.md`、`requirement.md`
- **变更相关文件**：`change.md`、`plan.md`、`design.md`、`handoff.md`
- **角色与经验文件**：当前角色文件、相关经验文件
- **产出文件**：生成的代码、文档、配置文件

#### 2.2.5 接收方指引
- **加载步骤**：6步加载流程，确保正确恢复上下文
- **继续工作的步骤**：验证、继续、报告的3步流程
- **注意事项**：未完成任务、已知风险、特别关注、限制条件

#### 2.2.6 元数据
- **协议版本**：当前模板版本
- **创建时间**：文件创建时间
- **最后更新时间**：最后修改时间
- **相关变更**：引用的相关变更（chg-01、chg-02、chg-05）
- **文件校验**：完整性、可访问性、准确性状态

### 2.3 可选字段说明

#### 2.3.1 会话记忆摘要
- **关键对话片段**：选择最重要的对话内容进行摘要
- **重要发现**：在任务执行过程中发现的重要信息
- **经验教训**：可以泛化的经验或教训

#### 2.3.2 联系方式
- **交接方标识**：更详细的标识信息
- **问题反馈**：如何反馈交接中的问题
- **紧急联系**：紧急情况下的联系方式

## 3. 交接内容规范

### 3.1 必须传递的文件列表

#### 3.1.1 核心状态文件（必须）
1. **`.workflow/state/runtime.yaml`**
   - 包含：`current_requirement`、`stage`、`conversation_mode`
   - 用途：恢复运行时状态
   - 验证：确保字段值正确反映当前状态

2. **`.workflow/state/sessions/[req-XX]/[chg-XX]/session-memory.md`**
   - 包含：执行日志、产出文件、经验沉淀、待处理问题
   - 用途：了解执行历史和当前状态
   - 验证：检查文件是否存在且内容完整

3. **`.workflow/flow/requirements/[req-XX-需求标题]/requirement.md`**
   - 包含：需求背景、目标、验收标准
   - 用途：理解任务背景和目标
   - 验证：确保是最新版本

#### 3.1.2 变更相关文件（必须）
1. **`change.md`**
   - 包含：变更标题、目标、范围、不包含内容
   - 用途：了解变更边界和约束
   - 验证：检查与当前任务的一致性

2. **`plan.md`**
   - 包含：执行步骤、预期产物、验证方法
   - 用途：指导后续工作
   - 验证：确认剩余步骤和已完成步骤

3. **`design.md`**（如有）
   - 包含：设计说明、实现细节
   - 用途：理解技术实现
   - 验证：检查设计完整性

4. **`handoff.md`**（本文件）
   - 包含：交接协议内容
   - 用途：交接本身
   - 验证：检查必填字段完整性

#### 3.1.3 角色与经验文件（按需）
1. **角色文件**：`.workflow/context/roles/[stage].md`
   - 根据当前 `stage` 加载
   - 用途：了解角色约束和行为规范

2. **经验文件**：
   - `.workflow/context/experience/stage/[stage].md`
   - `.workflow/context/experience/tool/harness.md`
   - 用途：避免重复错误，提高效率

#### 3.1.4 产出文件（按需）
- 生成的代码文件
- 生成的文档文件
- 配置文件
- 测试文件

### 3.2 状态信息规范

#### 3.2.1 当前 stage 和进度描述
- **阶段描述格式**：`[stage]`（如 `executing`、`testing`）
- **进度评估标准**：
  - `刚开始`：计划制定完成，尚未开始实质性工作
  - `进行中`：已开始执行，有部分成果但未完成
  - `接近完成`：主要工作已完成，剩余收尾工作
  - `已完成`：所有工作完成，等待验证或验收
- **进度标识**：使用百分比或完成步骤数/总步骤数

#### 3.2.2 已完成工作记录
- **格式**：使用 Markdown 复选框 `[x]`
- **内容**：具体、可验证的工作项
- **组织**：按步骤或功能模块分组
- **验证**：每个工作项应有对应的产出或证据

#### 3.2.3 待处理问题标识
- **问题分类**：
  - 技术问题：实现难点、技术限制
  - 流程问题：流程阻塞、依赖缺失
  - 资源问题：时间、人力、工具限制
- **状态标识**：
  - `未开始`：尚未调查
  - `调查中`：正在分析
  - `已定位`：原因已找到
  - `解决中`：正在实施解决方案
- **优先级**：`高`、`中`、`低`

### 3.3 会话记忆摘要规范

#### 3.3.1 关键对话片段提取
- **提取原则**：
  - 与核心决策相关的对话
  - 揭示重要技术选择的对话
  - 包含关键约束或限制的对话
  - 记录重要发现或突破的对话
- **避免提取**：
  - 常规的技术讨论细节
  - 已记录到文件的信息
  - 无关的闲聊或确认

#### 3.3.2 重要决策总结
- **决策要素**：
  - 决策内容：具体的选择
  - 决策背景：为什么需要决策
  - 备选方案：考虑过的其他选项
  - 选择理由：为什么选择这个方案
  - 预期影响：对后续工作的影响
- **总结格式**：简洁明了，突出重点

#### 3.3.3 避免信息过载的方法
1. **摘要而非复制**：总结要点，不复制完整对话
2. **结构化组织**：按主题或时间组织
3. **重要性排序**：最重要的信息放在前面
4. **引用而非包含**：引用文件内容，不重复包含
5. **定期清理**：只保留当前阶段相关的信息

## 4. 接收方指引规范

### 4.1 加载交接内容的步骤

#### 4.1.1 步骤1：读取 handoff.md
- **目的**：了解任务状态和交接内容
- **操作**：完整阅读 handoff.md 文件
- **验证**：理解摘要、任务状态、关键决策
- **输出**：对任务有整体认识

#### 4.1.2 步骤2：恢复状态文件
- **目的**：恢复运行时状态
- **操作**：
  1. 检查 `runtime.yaml` 是否存在
  2. 验证 `current_requirement`、`stage` 正确性
  3. 如有必要，更新 `locked_requirement`、`locked_stage`
- **验证**：状态文件内容与 handoff.md 描述一致
- **输出**：正确的运行时状态

#### 4.1.3 步骤3：加载角色文件
- **目的**：了解当前角色约束
- **操作**：根据 `stage` 加载对应的角色文件
- **验证**：角色文件存在且可读
- **输出**：明确角色职责和行为约束

#### 4.1.4 步骤4：加载经验文件
- **目的**：避免重复错误
- **操作**：根据 `stage` 加载相关经验文件
- **验证**：经验文件存在且相关
- **输出**：了解相关经验和教训

#### 4.1.5 步骤5：加载变更文档
- **目的**：了解变更详情
- **操作**：读取 `change.md`、`plan.md`、`design.md`
- **验证**：文档完整且与 handoff.md 一致
- **输出**：理解变更范围和执行计划

#### 4.1.6 步骤6：恢复会话记忆
- **目的**：了解执行历史
- **操作**：读取 `session-memory.md`
- **验证**：会话记忆完整且更新到最新
- **输出**：了解已完成的步骤和当前状态

### 4.2 注意事项清单

#### 4.2.1 未完成的任务项
- **识别**：检查 handoff.md 中的 `待完成的工作` 列表
- **优先级**：根据任务依赖性和重要性排序
- **资源**：评估完成所需的时间、工具、知识
- **计划**：制定继续工作的具体计划

#### 4.2.2 已知风险和限制
- **技术风险**：实现难点、技术债务
- **流程风险**：依赖缺失、流程阻塞
- **资源风险**：时间压力、工具限制
- **质量风险**：测试覆盖、代码质量
- **应对策略**：针对每个风险的缓解措施

#### 4.2.3 需要特别关注的问题
- **关键路径**：影响整体进度的关键问题
- **技术债务**：需要后续处理的技术问题
- **外部依赖**：依赖的外部系统或资源
- **质量门禁**：必须满足的质量标准
- **监控重点**：需要特别监控的指标

### 4.3 继续工作的步骤

#### 4.3.1 步骤1：验证交接完整性
- **文件检查**：所有必须传递的文件是否存在且可读
- **状态验证**：任务状态描述是否准确
- **决策验证**：关键决策记录是否完整
- **指引验证**：接收方指引是否清晰可行
- **问题记录**：发现的问题记录到 session-memory

#### 4.3.2 步骤2：从交接点继续
- **任务选择**：从 `待完成的工作` 列表中选择下一个任务
- **计划遵循**：遵循 `plan.md` 中的剩余步骤
- **决策参考**：参考 `关键决策记录` 中的决策
- **进度更新**：在 `session-memory.md` 中更新进度
- **问题处理**：处理 `未解决的问题` 列表中的问题

#### 4.3.3 步骤3：报告交接状态
- **完成确认**：在 handoff.md 末尾的 `交接完成确认` 打勾
- **状态记录**：在 `session-memory.md` 中记录交接完成
- **问题反馈**：如有问题，记录并考虑反馈给交接方
- **经验沉淀**：如有交接相关的经验，记录到经验文件

## 6. 集成验证

### 6.1 与 chg-02 决策树的集成

#### 6.1.1 接口定义
- **触发条件**：chg-02 决策树输出"新开 agent"决策时
- **输入**：任务状态、关键决策、必须文件列表
- **输出**：符合本模板的 `handoff.md` 文件
- **流程**：决策 → 创建 handoff → 新开 agent → 加载 handoff

#### 6.1.2 规范要求
1. **必须使用本模板**：遵循 `handoff.md` 模板结构
2. **必须包含必填字段**：任务状态、关键决策、下一步动作
3. **必须验证完整性**：交接前验证所有必要信息已包含
4. **必须提供接收指引**：明确告知接收方如何继续

#### 6.1.3 验证方法
1. **模板一致性检查**：handoff 文件结构符合模板
2. **必填字段检查**：所有必填字段已填写
3. **路径有效性检查**：文件路径存在且可访问
4. **指引可行性检查**：接收方指引具体可行

### 6.2 与 chg-05 恢复路径的集成

#### 6.2.1 接口定义
- **恢复场景**：handoff 交接失败或出现问题时的恢复
- **输入**：有问题的 handoff 文件、错误信息
- **输出**：恢复措施、修正后的 handoff、问题报告
- **流程**：发现问题 → 分析原因 → 实施恢复 → 验证恢复

#### 6.2.2 恢复路径
1. **文件缺失恢复**：缺失文件的恢复措施
2. **信息不完整恢复**：补充缺失信息的措施
3. **路径错误恢复**：修正文件路径的措施
4. **状态不一致恢复**：修正状态信息的措施
5. **指引不可行恢复**：修正接收方指引的措施

#### 6.2.3 验证方法
1. **恢复场景模拟**：模拟各种交接问题场景
2. **恢复措施验证**：验证恢复措施的有效性
3. **恢复后验证**：验证恢复后的 handoff 可用性
4. **问题记录验证**：验证问题记录和报告

### 6.3 使用指南

#### 6.3.1 创建 handoff 的步骤
1. **准备阶段**：
   - 确认需要交接（根据 chg-02 决策或用户请求）
   - 收集所有必要信息
   - 确保重要信息已保存到文件

2. **填写阶段**：
   - 复制 `handoff.md` 模板
   - 按章节填写所有必填字段
   - 根据需要填写可选字段
   - 验证文件路径和内容准确性

3. **验证阶段**：
   - 检查所有必填字段是否完整
   - 验证文件路径是否存在且可访问
   - 确保接收方指引清晰可行
   - 进行完整性自检

4. **交接阶段**：
   - 保存 handoff 文件
   - 确保所有必须传递的文件可访问
   - 如有必要，压缩或打包相关文件
   - 准备新开 agent

#### 6.3.2 使用 handoff 的步骤
1. **接收阶段**：
   - 在新 agent 中读取 handoff.md
   - 按照"加载步骤"恢复上下文
   - 验证交接完整性

2. **继续阶段**：
   - 按照"继续工作的步骤"继续任务
   - 处理待完成的工作
   - 参考关键决策记录

3. **完成阶段**：
   - 标记交接完成确认
   - 在 session-memory 中记录交接
   - 如有经验，沉淀到经验文件

#### 6.3.3 最佳实践
1. **定期更新**：在任务关键节点更新 handoff

### 8.2 常见问题解答

**Q1：什么时候需要创建 handoff 文件？**
A1：当 chg-02 决策树建议新开 agent，或用户主动要求交接时。

**Q2：handoff 文件应该放在哪里？**
A2：放在当前变更目录下，与 change.md、plan.md 同级。

**Q3：如何确保 handoff 文件的完整性？**
A3：按照本设计文档的验证标准进行检查，特别是必填字段完整性。

**Q4：如果交接过程中发现问题怎么办？**
A4：记录到 session-memory.md 的待处理问题，并参考 chg-05 的恢复路径。

**Q5：handoff 文件需要版本控制吗？**
A5：建议进行版本控制，特别是长期任务可能需要多次交接。
