# 执行计划

## 依赖关系
- **前置依赖**：chg-01（监控指标）、chg-02（决策树）、chg-03（handoff 协议）
- **后置依赖**：为 chg-06 提供风险框架

## 执行步骤

### 步骤 1：分析上下文爆炸风险
1. 调研上下文爆炸的实际影响：
   - API 调用失败的具体表现
   - 工作流中断的后果
   - 数据丢失的风险
2. 收集历史教训和最佳实践
3. 定义风险等级和影响范围

### 步骤 2：注册风险条目
1. 更新 `constraints/risk.md`：
   - 新增风险条目 "上下文爆炸导致工作流中断"
   - **风险描述**：详细描述上下文长度超限的触发条件、表现、影响
   - **影响分析**：会话丢失、任务中断、需要重新开始等后果
   - **缓解措施**：引用本需求设计的监控指标、阈值、维护动作
   - **检测方法**：如何识别上下文爆炸风险
2. 确保风险条目格式规范、内容完整

### 步骤 3：设计恢复路径
1. 更新 `constraints/recovery.md`：
   - 新增 "上下文超限恢复" 章节
   - **检测阶段**：如何发现上下文已超限
   - **决策阶段**：根据 chg-02 决策树选择恢复动作
   - **执行阶段**：执行维护动作的具体步骤
   - **验证阶段**：验证恢复是否成功、工作流是否可继续
   - **handoff 集成**：如需新开 agent，引用 chg-03 handoff 协议
2. 设计分层恢复策略：
   - 轻度超限：使用 `/compact`
   - 中度超限：使用 `/clear`
   - 严重超限：新开 agent + handoff

### 步骤 4：更新行为边界
1. 更新 `constraints/boundaries.md`：
   - 增加与上下文维护相关的行为边界
   - **主 agent 职责**：定期监控上下文负载、触发维护动作
   - **subagent 职责**：报告上下文消耗、建议清理
   - **用户职责**：关注上下文状态、及时响应提醒
   - **工具使用边界**：何时可以使用 `/compact`、`/clear` 等命令
2. 定义边界违反的处理方式

### 步骤 5：集成验证
1. 验证与 chg-01 的集成：
   - 风险条目引用监控指标
   - 恢复路径使用阈值设计
2. 验证与 chg-02 的集成：
   - 恢复决策引用决策树
   - 动作选择逻辑一致
3. 验证与 chg-03 的集成：
   - handoff 协议在恢复路径中的使用
   - 交接流程的完整性
4. 创建端到端恢复场景测试用例

### 步骤 6：创建使用指南
1. 设计风险识别指南：
   - 如何识别上下文爆炸的早期迹象
   - 如何评估风险等级
   - 何时触发恢复流程
2. 设计恢复操作指南：
   - 逐步恢复操作说明
   - 常见问题解决方法
   - 成功恢复的验证标准

## 预期产物
1. 更新的 `constraints/risk.md`：新增风险条目
2. 更新的 `constraints/recovery.md`：新增恢复路径
3. 更新的 `constraints/boundaries.md`：新增行为边界
4. `design.md`：集成设计说明和验证用例

## 验证方法
1. 人工评审风险条目的完整性和准确性
2. 模拟恢复场景验证恢复路径的可行性
3. 检查与前置变更的引用关系是否正确
4. 验证行为边界的合理性和可执行性

## 注意事项
1. 风险描述要具体、可识别，避免模糊表述
2. 恢复路径要详细、可操作，避免理论化
3. 行为边界要明确、可执行，避免歧义
4. 确保与工作流其他约束的一致性
5. 考虑实际使用中的边缘情况和异常处理
