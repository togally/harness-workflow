# 执行计划

## 依赖关系
- **前置依赖**：chg-01（监控指标与阈值设计）
- **后置依赖**：为 chg-04、chg-05、chg-06 提供决策逻辑

## 执行步骤

### 步骤 1：分析维护动作特性
1. 研究 `/compact` 命令：
   - 工作原理：如何压缩上下文
   - 适用场景：何时使用效果最佳
   - 限制：压缩后可能丢失的信息类型
2. 研究 `/clear` 命令：
   - 工作原理：如何清空上下文
   - 适用场景：何时必须使用
   - 风险：清空后无法恢复的信息
3. 研究新开 agent 流程：
   - 如何创建新 agent
   - 交接要求：需要传递哪些信息
   - 与 handoff 协议（chg-03）的集成

### 步骤 2：设计决策树输入输出
1. 定义决策树输入：
   - 当前上下文负载（token 估算百分比）
   - 消息条数
   - 文件读取次数
   - 会话时长
   - 当前任务状态（阶段、进度）
2. 定义决策树输出：
   - 建议的维护动作（`/compact`、`/clear`、新开 agent）
   - 动作优先级
   - 预期效果说明

### 步骤 3：创建决策逻辑
1. 基于 chg-01 的阈值设计决策规则：
   - **预警阈值范围**：建议使用 `/compact`，继续当前任务
   - **强制维护阈值**：必须执行维护动作，根据任务状态选择 `/compact` 或 `/clear`
   - **紧急阈值**：必须新开 agent，使用 handoff 协议交接
2. 考虑任务状态因素：
   - 任务刚开始 vs 任务接近完成
   - 关键决策是否已记录
   - 是否有未保存的重要信息
3. 设计决策流程图或决策表

### 步骤 4：定义各动作适用场景
1. `/compact` 适用场景详细定义：
   - 历史消息较多但仍有压缩空间
   - 任务尚未完成，需要保留部分上下文
   - 操作步骤和注意事项
2. `/clear` 适用场景详细定义：
   - 历史消息已无效或干扰
   - 需要全新上下文重新开始
   - 操作步骤和风险提示
3. 新开 agent 适用场景详细定义：
   - 上下文已爆满，无法继续
   - 触发条件和交接要求
   - handoff 协议引用

### 步骤 5：集成与验证
1. 与 chg-01 阈值设计的集成验证
2. 与 chg-03 handoff 协议的接口定义
3. 创建验证用例

## 预期产物
1. `design.md`：包含完整的决策树设计、适用场景定义
2. 决策流程图（可选）
3. 验证用例文档

## 验证方法
1. 人工评审决策逻辑的合理性
2. 模拟不同场景验证决策结果
3. 检查与 chg-01 阈值的集成是否一致
4. 验证与 chg-03 handoff 协议的接口是否清晰

## 注意事项
1. 决策逻辑要简单明确，避免过度复杂
2. 考虑实际使用中的边缘情况
3. 为每个维护动作提供明确的操作指引
4. 确保与后续变更的接口定义清晰
