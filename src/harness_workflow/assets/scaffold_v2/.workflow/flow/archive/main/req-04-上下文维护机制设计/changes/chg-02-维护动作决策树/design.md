# 维护动作决策树设计

## 1. 概述

本文档定义 harness-workflow 系统的维护动作决策逻辑，根据当前上下文负载和阈值，决定使用 `/compact`、`/clear` 还是新开 agent。

**设计目标**：
- 基于 chg-01 的监控指标和阈值，提供明确的维护动作决策逻辑
- 定义各维护动作的适用场景和操作指引
- 为 chg-03 handoff 协议、chg-04 工具层集成提供接口
- 确保决策逻辑简单明确，易于理解和执行

**适用范围**：所有使用 Claude Code 作为执行平台的 harness-workflow 会话。

## 2. 决策树输入

### 2.1 主要输入指标（来自 chg-01）

| 指标 | 定义 | 量化方法 |
|------|------|----------|
| **Token 估算百分比** | 当前上下文占用的 token 数量与最大限制（102400 tokens）的百分比 | 基于消息历史、文件内容、工具输出的估算公式 |
| **消息条数** | 用户消息 + AI 回复的总条数 | 实时统计，每次消息交互后更新 |
| **文件读取次数** | Read 工具调用次数 | 按文件类型分类统计 |
| **会话时长** | 从会话开始到当前的时间 | 持续计时，定期更新 |

### 2.2 辅助输入：任务状态

| 状态维度 | 取值 | 说明 |
|----------|------|------|
| **任务阶段** | `requirement_review`、`planning`、`executing`、`testing`、`acceptance`、`regression`、`done` | 当前工作流阶段 |
| **任务进度** | `刚开始`、`进行中`、`接近完成`、`已完成` | 相对进度评估 |
| **关键决策记录** | `已记录`、`部分记录`、`未记录` | 关键决策是否已保存到文件 |
| **未保存信息** | `有重要信息`、`无重要信息` | 是否有未保存到文件的重要信息 |

### 2.3 阈值定义（引用 chg-01）

#### 2.3.1 主要阈值（基于 Token 估算百分比）
- **预警阈值**：70-80% 最大上下文（71680-81920 tokens）
- **强制维护阈值**：85-90% 最大上下文（87040-92160 tokens）
- **紧急阈值**：>95% 最大上下文（>97280 tokens）

#### 2.3.2 辅助阈值
- **消息条数阈值**：
  - 预警：>100 条消息
  - 强制维护：>150 条消息  
  - 紧急：>200 条消息
- **文件读取次数阈值**：
  - 预警：>50 次读取
  - 强制维护：>80 次读取
  - 紧急：>100 次读取
- **会话时长阈值**：
  - 预警：>2 小时
  - 强制维护：>4 小时
  - 紧急：>6 小时

## 3. 决策逻辑

### 3.1 决策流程

```
开始
  ↓
检查当前上下文负载
  ↓
判断是否达到阈值
  ├── 未达到阈值 → 继续当前任务
  ├── 达到预警阈值 → 建议执行维护动作
  ├── 达到强制维护阈值 → 必须执行维护动作
  └── 达到紧急阈值 → 必须立即处理
  ↓
评估任务状态
  ├── 任务刚开始 → 倾向 `/clear`
  ├── 任务进行中 → 倾向 `/compact`
  ├── 任务接近完成 → 强烈倾向 `/compact`
  └── 任务已完成 → 倾向 `/clear`
  ↓
选择维护动作
  ├── `/compact`：保留关键信息，压缩细节
  ├── `/clear`：清空历史，重新开始
  └── 新开 agent：交接任务，重新开始
  ↓
提供操作指引
  ↓
结束
```

### 3.2 决策表

| 阈值级别 | 任务状态 | 建议动作 | 优先级 | 预期效果 |
|----------|----------|----------|--------|----------|
| **预警阈值** | 任何状态 | `/compact` | 建议（可延迟） | 压缩历史消息，保留关键信息，继续当前任务 |
| **强制维护阈值** | 任务刚开始 | `/clear` | 必须（立即执行） | 清空历史，重新加载必要文件，从头开始 |
| **强制维护阈值** | 任务进行中 | `/compact` | 必须（立即执行） | 压缩历史，保留关键决策和状态，继续工作 |
| **强制维护阈值** | 任务接近完成 | `/compact` | 必须（立即执行） | 压缩历史，保留完成状态和关键信息，完成收尾 |
| **强制维护阈值** | 任务已完成 | `/clear` | 必须（立即执行） | 清空历史，准备下一个任务 |
| **紧急阈值** | 任何状态 | **新开 agent** | 紧急（立即处理） | 创建 handoff 交接，在新会话中继续任务 |

### 3.3 特殊情况处理

#### 3.3.1 关键决策未记录
- **场景**：达到强制维护阈值，但关键决策未保存到文件
- **处理**：
  1. 先记录关键决策到 `session-memory.md` 或相关文件
  2. 再执行维护动作（优先 `/compact` 或新开 agent）
  3. 避免使用 `/clear`，除非已确保所有重要信息已保存

#### 3.3.2 多指标同时触发
- **场景**：Token 估算达到预警阈值，同时消息条数达到强制维护阈值
- **处理**：按**最严重的阈值**处理
  - 任一指标达到紧急阈值 → 按紧急阈值处理
  - 任一指标达到强制维护阈值 → 按强制维护阈值处理
  - 所有指标都在预警范围内 → 按预警阈值处理

#### 3.3.3 阶段转换时
- **场景**：从 `executing` 阶段转换到 `testing` 阶段
- **处理**：
  - 如果上下文负载较高（>60%），建议执行 `/compact`
  - 如果历史上下文与新阶段无关，建议执行 `/clear`
  - 确保新阶段开始时有干净的上下文

## 4. 维护动作适用场景

### 4.1 `/compact` 命令

#### 4.1.1 适用场景
1. **历史消息较多但仍有压缩空间**
   - 消息条数：100-200 条
   - 文件读取次数：50-100 次
   - 会话时长：2-6 小时

2. **任务尚未完成，需要保留部分上下文**
   - 当前阶段：`executing`、`testing`、`acceptance`
   - 任务进度：`进行中`、`接近完成`
   - 关键决策：`已记录` 或 `部分记录`

3. **需要保留关键信息但减少细节**
   - 保留：当前 stage、requirement、关键决策、待处理问题
   - 压缩：详细实现步骤、历史对话细节、已完成的代码片段

#### 4.1.2 操作步骤
1. 执行 `/compact` 命令
2. 重新读取必要文件：
   - `.workflow/state/runtime.yaml`
   - 当前 `session-memory.md`
   - 当前变更的 `change.md`、`plan.md`、`design.md`
   - 其他必要配置文件
3. 验证关键信息是否保留
4. 如有遗漏，补充读取相关文件

#### 4.1.3 注意事项
- 压缩后可能丢失一些细节信息
- 需确保关键决策和状态信息已保留
- 如有不确定，可先尝试 `/compact`，观察效果
- 压缩后如发现问题，可考虑进一步维护

### 4.2 `/clear` 命令

#### 4.2.1 适用场景
1. **历史消息已无效或干扰**
   - 消息历史与当前任务完全无关
   - 上下文混乱，影响新任务执行
   - 需要彻底重新开始

2. **任务切换或阶段转换**
   - 从 `done` 阶段进入新 `requirement_review`
   - 从 `regression` 切换回正常开发流程
   - 需要全新上下文开始新任务

3. **简单任务或快速验证**
   - 任务简单，不需要复杂历史
   - 快速验证某个功能或修复
   - 测试环境清理

#### 4.2.2 操作步骤
1. **确保重要信息已保存**：
   - 关键决策记录到文件
   - 任务状态保存到 `session-memory.md`
   - 必要文件已更新
2. 执行 `/clear` 命令
3. 重新加载所有必要文件：
   - `.workflow/state/runtime.yaml`
   - `.workflow/context/index.md`
   - 当前角色文件（如 `executing.md`）
   - 相关经验文件
   - 当前变更的所有文档
4. 重新建立上下文

#### 4.2.3 风险提示
- **所有历史信息丢失**：无法恢复之前的对话历史
- **需要重新加载文件**：增加初始设置时间
- **可能遗漏信息**：如果未提前保存关键信息
- **适用性有限**：不适用于复杂、进行中的任务

### 4.3 新开 agent

#### 4.3.1 适用场景
1. **上下文已爆满，无法继续**
   - Token 估算 >95% 最大上下文
   - 消息条数 >200 条
   - 文件读取次数 >100 次
   - 会话时长 >6 小时

2. **任务必须继续，但无法压缩或清空**
   - 任务复杂，需要完整历史
   - 关键决策多，无法简单压缩
   - 任务接近完成，不能重新开始

3. **系统性能问题**
   - 响应速度明显下降
   - 工具调用失败率增加
   - 内存或性能问题

#### 4.3.2 触发条件
1. **紧急阈值触发**：任一主要指标达到紧急阈值
2. **性能问题**：明显的系统性能下降
3. **用户请求**：用户主动要求新开 agent

#### 4.3.3 交接要求
必须创建 `handoff.md` 文件，包含以下内容：

1. **任务状态摘要**：
   - 当前 requirement 和 change
   - 当前 stage 和进度
   - 已完成的工作
   - 待完成的工作

2. **关键决策记录**：
   - 重要技术决策
   - 设计选择理由
   - 已解决的问题
   - 未解决的问题

3. **必须传递的文件**：
   - `session-memory.md`
   - 当前变更的 `change.md`、`plan.md`、`design.md`
   - 相关配置文件
   - 生成的代码或文档

4. **接收方指引**：
   - 如何加载交接内容
   - 下一步动作
   - 注意事项和风险提示
   - 联系方式（如有）

#### 4.3.4 handoff 协议引用
必须遵循 chg-03 设计的 `handoff.md` 协议模板，确保交接标准化。

## 5. 集成验证

### 5.1 与 chg-01 阈值设计的集成

#### 5.1.1 直接引用
- 使用 chg-01 定义的所有阈值
- 继承相同的指标定义和量化方法
- 保持阈值一致性

#### 5.1.2 扩展应用
- 将阈值应用于决策逻辑
- 考虑多指标组合情况
- 添加任务状态维度

#### 5.1.3 验证方法
1. 模拟不同阈值场景
2. 验证决策结果符合预期
3. 检查阈值引用准确性

### 5.2 与 chg-03 handoff 协议的接口

#### 5.2.1 接口定义
- **触发条件**：新开 agent 决策必须触发 handoff 协议
- **输入**：任务状态、关键决策、必须文件列表
- **输出**：符合规范的 `handoff.md` 文件
- **流程**：决策 → 创建 handoff → 新开 agent → 加载 handoff

#### 5.2.2 规范要求
1. **必须使用标准模板**：遵循 chg-03 的 `handoff.md` 模板
2. **必须包含所有必填字段**：任务状态、关键决策、下一步动作
3. **必须验证完整性**：交接前验证所有必要信息已包含
4. **必须提供接收指引**：明确告知接收方如何继续

#### 5.2.3 验证方法
1. 检查 handoff 文件结构符合模板
2. 验证必填字段完整性
3. 模拟交接流程

### 5.3 验证用例

#### 用例1：预警阈值触发
- **模拟条件**：Token 估算 75%，消息条数 120，任务进行中
- **预期决策**：建议 `/compact`，优先级：建议
- **验证点**：决策符合预警阈值逻辑

#### 用例2：强制维护阈值 + 任务刚开始
- **模拟条件**：Token 估算 88%，消息条数 160，任务刚开始
- **预期决策**：必须 `/clear`，优先级：必须
- **验证点**：考虑任务状态因素

#### 用例3：强制维护阈值 + 任务接近完成
- **模拟条件**：Token 估算 87%，消息条数 170，任务接近完成，关键决策已记录
- **预期决策**：必须 `/compact`，优先级：必须
- **验证点**：任务进度影响决策

#### 用例4：紧急阈值触发
- **模拟条件**：Token 估算 96%，消息条数 210，任务进行中
- **预期决策**：必须新开 agent + handoff，优先级：紧急
- **验证点**：紧急阈值触发正确动作

#### 用例5：多指标冲突
- **模拟条件**：Token 估算 72%（预警），消息条数 180（强制维护）
- **预期决策**：按最严重阈值处理 → 必须执行维护动作
- **验证点**：多指标处理逻辑正确

#### 用例6：关键决策未记录
- **模拟条件**：强制维护阈值，任务进行中，关键决策未记录
- **预期决策**：先记录关键决策，再执行 `/compact`
- **验证点**：特殊情况处理正确

## 6. 与后续变更的接口

### 6.1 chg-03 handoff 交接协议
- **输入**：本决策树的新开 agent 决策
- **输出**：标准化的 handoff 交接文件
- **接口**：决策 → handoff 创建 → 新会话加载

### 6.2 chg-04 工具层集成
- **输入**：维护动作决策结果
- **输出**：工具调用规范（如何执行 `/compact`、`/clear`）
- **接口**：决策 → 工具调用 → 效果验证

### 6.3 chg-05 风险与恢复路径
- **输入**：维护动作可能的风险场景
- **输出**：风险应对和恢复措施
- **接口**：决策 → 风险评估 → 恢复准备

### 6.4 chg-06 经验沉淀与角色职责更新
- **输入**：决策树的实际使用经验
- **输出**：角色职责中关于上下文维护的部分
- **接口**：经验收集 → 职责更新 → 流程优化

## 7. 实施建议

### 7.1 决策执行流程
1. **定期检查**：主 agent 定期检查上下文负载
2. **阈值判断**：根据 chg-01 阈值判断当前级别
3. **状态评估**：评估任务状态和进度
4. **决策选择**：根据决策表选择维护动作
5. **执行动作**：执行选定的维护动作
6. **效果验证**：验证维护动作效果

### 7.2 用户交互
1. **透明通知**：告知用户上下文状态和维护建议
2. **确认执行**：重要维护动作需用户确认
3. **进度报告**：报告维护动作执行情况和效果
4. **问题处理**：如维护失败，提供恢复选项

### 7.3 监控优化
1. **经验收集**：记录每次维护决策的效果
2. **阈值调整**：根据实际使用调整阈值
3. **决策优化**：优化决策逻辑和规则
4. **工具改进**：改进维护工具和流程

### 8.2 快速参考表

| 情况 | 动作 | 优先级 | 说明 |
|------|------|--------|------|
| Token 70-80% | `/compact` | 建议 | 压缩历史，继续工作 |
| Token 85-90% + 任务刚开始 | `/clear` | 必须 | 清空，重新开始 |
| Token 85-90% + 任务进行中 | `/compact` | 必须 | 压缩，保留关键信息 |
| Token >95% | 新开 agent | 紧急 | 交接，新会话继续 |
| 消息 >200 条 | 新开 agent | 紧急 | 消息过多，需交接 |
| 关键决策未记录 | 先记录再维护 | 必须 | 避免信息丢失 |

### 8.3 注意事项
1. **安全第一**：确保重要信息已保存再执行维护
2. **用户确认**：重要维护动作需用户确认
3. **效果验证**：维护后验证关键信息保留
4. **记录经验**：记录维护决策和效果，用于优化
5. **保持简单**：决策逻辑应简单明确，易于执行
