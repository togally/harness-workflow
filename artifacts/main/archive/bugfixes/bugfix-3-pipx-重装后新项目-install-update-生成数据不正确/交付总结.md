# 交付总结：bugfix-3 pipx 重装后新项目 install/update 生成数据不正确

## 需求是什么
用户 pipx reinstall harness-workflow 之后，在已存在的目标项目（`/Users/jiazhiwei/claudeProject/PetMallPlatform`）跑 `harness install` + `harness update`，感知"生成的数据不对"——模板新章节（req-26 "对人文档"等）追不上最新版本，磁盘上 `legacy-cleanup/.workflow/context/experience/index.md-N` 持续累积。

## 交付了什么
- **代码修复**（`src/harness_workflow/workflow_helpers.py`，+16/-1，2 处改动）：
  - 根因 A — `_sync_requirement_workflow_managed_files` 新增 `adopt-as-managed` 分支（漏登记 hash 视为可信任覆盖；用户自定义文件 hash 已登记时保持 skip）。
  - 根因 B — `LEGACY_CLEANUP_TARGETS` 移除 `.workflow/context/experience/index.md`，终止"搬家→重建→再搬家"循环。
- **回归用例**（`tests/test_workflow_helpers_update_idempotent.py`）：两条红用例锁死根因 A、B，防止未来退化。
- **对人文档全量**：`回归简报.md`（问题/根因/路由）+ `测试结论.md`（用例设计）+ `实施说明.md`（修复摘要）+ `验收摘要.md`（核对结果）+ 本《交付总结.md》，`harness validate --human-docs` 5/5 ✓。
- **5 条衍生问题转 sug 池**：D1~D5 分别对应多生成器 hash 竞争、adopt 误覆盖风险、对人文档自检契约、stage_timestamps 盲区、CLI cwd 敏感。

## 结果是什么
- **验收 pass**：bugfix.md Validation Criteria 6/6 实质通过；全量 `pytest` 146 passed / 50 skipped / 1 pre-existing failure（已独立 git stash 认证为与本 bugfix 无关）；临时副本烟测 stdout 不再出现 `skipped modified .../context/roles/*.md` 和 `archived legacy .../experience/index.md`。
- **影响面**：修复覆盖所有"历史 install 后 scaffold 演进"的目标项目；下次 update 会把所有漏登记的模板文件一次性 adopt。用户项目（PetMallPlatform）的代码未被本 bugfix 修改，用户需要自己跑 `harness update` 才会生效。
- **遗留**：第二次 update 仍有 3 条 `skipped modified`（`experience/index.md` / `runtime.yaml` / `.codex/skills/harness/SKILL.md`，源自多生成器共享文件 hash 竞争），不影响本次修复的有效性；已转 sug-13 跟踪。

## 后续建议
1. 用户可一次性手动清理目标项目已堆积的 `legacy-cleanup/.workflow/context/experience/index.md*` 旧副本（本 bugfix 不自动清理，避免越界）。
2. 推进 sug-13（多生成器 hash 竞争）修复，彻底消除 update 幂等性二阶段问题。
3. 推进 sug-15（对人文档自检契约），让 stage 角色在产出对人文档后当场 `harness validate` 自检，避免到 acceptance 才发现契约违规。
