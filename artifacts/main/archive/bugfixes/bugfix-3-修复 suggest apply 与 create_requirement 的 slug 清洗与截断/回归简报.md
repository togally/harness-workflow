# 回归简报：bugfix-3 修复 suggest apply 与 create_requirement 的 slug 清洗与截断

## 问题是什么
- `harness suggest --apply sug-08` 将建议正文全句（含 `/` + 150+ 字）当 title 直接拼进 Path，需求目录被拆成 3 级嵌套、state yaml 同步污染、runtime.yaml 被错配。
- 任何 `harness requirement "含/的 标题"` / `harness bugfix "含/的 标题"` 都可复现，非 suggest 专属。

## 根因是什么
- `workflow_helpers.py` 中 `apply_suggestion`（L3100-3103）取首行无截断当 title，再调 `create_requirement`。
- `create_requirement`（L3275/L3290）+ `create_bugfix`（L3337/L3375）直接 `f"{id}-{raw_title}"` 拼 Path，**未调 `slugify_preserve_unicode` 且无长度上限**。
- 对照 `create_change` / `create_regression` / `rename_*` 已正确走 slugify —— `create_*` 族两处漏补，属继承链遗漏。
- 附加缺口：`apply_suggestion` 成功后 sug 源文件仅翻转 frontmatter，未归档到 `flow/suggestions/archive/`。

## 路由决策
- 真实 CLI 缺陷 → 转 bugfix 修复（bugfix-3）。
- 修复前先回滚脏数据：删除 req-30 两处 3 级嵌套、runtime.yaml 回退 pre-apply 快照、sug-08 frontmatter 回 `pending`。
- reg-01（原回归诊断）cancel，备份于 `/tmp/reg-01-backup/`。

## 后续建议
- 在 executing 阶段引入 `_path_slug(title, max_len=60)` helper 作为 `slugify_preserve_unicode` + 长度上限 + `strip("-")` 的统一入口，所有 `create_*` 族都下沉到此 helper，避免再次漏补。
- 测试层以 TDD 先红再绿，覆盖 `/` 嵌套、长 title 截断、空 title id-only 回退、sug 归档 + frontmatter 翻转四类场景。
