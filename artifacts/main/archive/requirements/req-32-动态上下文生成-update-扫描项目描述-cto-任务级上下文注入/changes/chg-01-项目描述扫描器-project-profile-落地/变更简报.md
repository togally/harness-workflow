# 变更简报：chg-01（项目描述扫描器 + project-profile 落地）

## 变更名

- 新建 `ProjectScanner` 底座，产出 `.workflow/context/project-profile.md`，为后续 update 集成与 briefing 注入铺地。

## 解决什么问题

- 覆盖 req-32（动态上下文生成：update 扫描项目描述 + CTO 任务级上下文注入）的 AC-01：让 `.workflow/context/` 第一次真正"知道"本仓是什么项目（语言 / 包名 / 主要依赖 / 入口命令 / 技术栈标签），并预留 LLM 填充项目用途与规范的占位 section。

## 怎么做

- 新增 `src/harness_workflow/project_scanner.py`：识别 pyproject.toml / package.json / pom.xml / go.mod / Cargo.toml / README / CLAUDE.md / AGENTS.md，静态解析为结构化字段。
- 提供 `build_project_profile(root)` / `render_project_profile(profile)` / `load_project_profile(path)` 三个入口；渲染产物含 frontmatter（`generated_at` + `content_hash`）+ 三段（结构化字段 / `## 项目用途（LLM 填充）` / `## 项目规范（LLM 填充）`）。
- content_hash 复用 `workflow_helpers._managed_hash`，保证与 update 侧 hash 口径一致。
- 不新建 prompts/ 目录；LLM 占位以 profile section 形式出现。
- TDD ≥ 12 条用例：每种描述文件 1 条 + 渲染快照 + hash 稳定 + 空项目兜底。

## 影响范围

- 新增：`src/harness_workflow/project_scanner.py`、`tests/test_project_scanner.py`。
- 不修改既有文件；不接入 CLI 命令。

## 预期验证

- `pytest tests/test_project_scanner.py -v` 全绿。
- 手动 `python -c "from harness_workflow.project_scanner import build_project_profile; from pathlib import Path; print(build_project_profile(Path('.')))"` 能看到 `package_name=harness-workflow` 与 python 技术栈标签。
- `pytest -q` 零回归。
- `harness validate --contract all` 绿。
