# 变更简报：chg-02（harness update 集成扫描器 + hash 漂移 + 用户自定义保护）

## 变更名

- 把 req-32（动态上下文生成：update 扫描项目描述 + CTO 任务级上下文注入）的 chg-01（项目描述扫描器 + project-profile 落地）产出的 scanner 接入 `harness update`，实现 profile 生成 / 漂移提示 / 用户自定义保护三件套。

## 解决什么问题

- 覆盖 req-32 的 AC-02 与 AC-05：每次 update 都自动刷新本仓 project-profile，技术栈演进可见；并保证用户自定义的 CLAUDE.md / AGENTS.md / SKILL.md 不会被模板覆盖。

## 怎么做

- `workflow_helpers.py::update_repo` 末尾新增 `_write_project_profile_if_changed`：对比新旧 profile 的 `content_hash`，一致则静默，不一致则 stderr 打印 `project-profile 已刷新（hash 漂移：<old8>→<new8>）` 并覆盖写。
- CLAUDE/AGENTS/SKILL 保护直接复用既有 `_is_user_authored`（req-31（批量建议合集（20条））的 chg-03（CLI/helper 剩余修复）的 sug-14（用户自定义 CLAUDE/AGENTS/SKILL 保护）），不新增判据；仅在跳过时补 stderr `skip user-authored: <relative>`（若原实现已打印则测试守护）。
- TDD ≥ 5 条：首次生成 / 两次 hash 稳定 / pyproject 改动漂移 / 用户自定义跳过 / 旧 profile 损坏兜底。

## 影响范围

- 修改：`src/harness_workflow/workflow_helpers.py`（约 50-70 行新增/修改）。
- 新增：`tests/test_update_project_profile.py`。
- 用户侧：`.workflow/context/project-profile.md` 首次出现并随 update 刷新。

## 预期验证

- 本仓连跑两次 `harness update`：第一次见 `generated .workflow/context/project-profile.md`，第二次 stderr 不再出现漂移提示。
- 编辑 `pyproject.toml` 的依赖版本后再跑一次，stderr 出现 `project-profile 已刷新（hash 漂移：...→...）`。
- 对 CLAUDE.md 做一处非模板修改后跑 update，该文件不被覆盖，stderr 出现 `skip user-authored: CLAUDE.md`。
- `pytest -q` 零回归；`harness validate --contract all` 绿。
