# 需求摘要：req-32（动态上下文生成：update 扫描项目描述 + CTO 任务级上下文注入）

## 目标

- 让 `harness update` 扫描本仓项目描述文件生成 `.workflow/context/project-profile.md`，并让 harness-manager / CTO 派发 subagent 时基于任务内容产出 `task_context_index` 建议清单，下游按索引最小加载，降低 context 消耗、提升命中率。

## 范围

- 包含：`harness update` 新增项目描述扫描（pyproject.toml / package.json / pom.xml / go.mod / Cargo.toml / README / CLAUDE.md / AGENTS.md）+ 静态解析结构化字段 + LLM prompt 占位
- 包含：`.workflow/context/project-profile.md` 落盘 + hash 漂移检测 + stderr 刷新提示
- 包含：harness-manager / CTO 派发 subagent 的 briefing 增加 `task_context_index` 字段（≤ 8 条，每条带理由），覆盖 `harness next --execute` / `ff --auto` / regression 派发路径
- 包含：每次派发的 `task_context_index` 落 `.workflow/state/sessions/{req-id}/task-context/{stage}-{seq}.md` 快照（审计 / 优化用），归档时随 req 迁入 `artifacts/.../sessions/`
- 不包含：自动改写 `experience/**` 或 `platforms.yaml`；不改造 tools-manager 推荐派发与 done 六层回顾派发；CLI 内不直接调 LLM
- 不包含：覆盖用户已自定义的 CLAUDE.md / AGENTS.md / SKILL.md（沿用 req-31（批量建议合集（20条））sug-14（用户自定义保护）机制）
- 不包含：CLI 观测命令（如 `harness status --context-stats`）—— 用户直接读快照文件

## 验收要点

- AC-01：本仓 `harness update` 后生成 `project-profile.md`，含结构化字段 + LLM 占位段 + 时间戳
- AC-02：二次 `harness update` hash 稳定；描述变更时 stderr 提示"project-profile 已刷新（hash 漂移）"
- AC-03：`harness next --execute` / `ff --auto` / regression 派发的 briefing 含 `task_context_index` 字段（≤ 8 条，`{path}: {reason}` 格式）
- AC-04：subagent 索引路径失效时不报错，回退到 `.workflow/context/index.md` 全量加载
- AC-05：用户自定义 CLAUDE.md / AGENTS.md 不被覆盖，stderr 提示跳过
- AC-06：TDD 红绿各一轮，`harness validate --contract all` 全绿
- AC-07：每次派发落 `.workflow/state/sessions/{req-id}/task-context/{stage}-{seq}.md` 快照（frontmatter + 正文），归档走既有路径随 req 迁移

## 风险

- R1 索引 backfire：CTO 塞过多索引反致 briefing 膨胀 → 每条带理由 + 硬上限 8 条，CLI 校验超限即截断并 warn
- R2 profile 漂移：技术栈演进后 profile 失效 → 每次 update 重扫对比 hash，差异时 stderr 提示刷新
