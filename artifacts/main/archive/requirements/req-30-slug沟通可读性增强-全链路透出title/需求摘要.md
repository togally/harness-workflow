# 需求摘要：req-30 slug 沟通可读性增强：全链路透出 title

## 目标

- 让任何人 / agent 在任何场景看到一个工作项（req / chg / sug / reg / bugfix）的引用时，都能不用额外跳转就知道它在说什么功能——以 `{id}（{title}）` 形态贯穿四层：汇报模板、CLI stdout、runtime/state yaml 字段、归档/索引。

## 范围

- **包含**：四层改造（L1 汇报模板、L2 CLI 渲染、L3 state yaml 冗余 `*_title` 字段、L4 归档/索引命名与元数据）；活跃工作项 title 回填；角色文件模板更新；测试覆盖与自证样本。
- **不做**：不重命名 slug / 不改路径结构；不回写历史 action-log / session-memory；不做 i18n；不引入外部元数据服务；不覆盖 legacy 目录。

## 验收要点

- **AC-01 / AC-02**：所有 agent 对人汇报、role 模板首次出现工作项时必须写 `{id}（{title}）`，裸 id 视为违规。
- **AC-03 / AC-04**：`harness status / next / ff / suggest` 等命令 stdout 默认带 title；title 缺失时降级打印 `(no title)` 不报错。
- **AC-05 / AC-06**：`runtime.yaml` 新增 `current_requirement_title` 等冗余字段；所有写 state 的路径同步写 title，新建缺 title 即失败。
- **AC-07 / AC-08**：归档目录 / `experience/index.md` 来源列在新增时必须含 title；历史条目按需回填。
- **AC-09 / AC-10**：至少 2 单测 + 1 smoke 覆盖渲染与写入主路径；req-30 自身文档作为"新约定"示范样本。

## 风险

- **缓存一致性**：runtime.yaml 的 `*_title` 与 state yaml 的 `title` 若写入路径不统一会错显——约定"state 权威 + runtime 缓存 + 读失败 fallback"规避。
- **模板退化**：agent 汇报仍可能漏写 title——由 done 阶段六层回顾 + `harness status` 轻量 lint 兜底。

## 候选方案（供用户选择）

- **A 轻量**（S，仅渲染/汇报层改，不动 state schema）——次选（不采纳）
- **B 双管（已选定 2026-04-21）**（M，渲染 + state 冗余字段 + 模板硬门禁，四层全覆盖）—— 采纳方案
- **C 激进**（L，title 升为 CLI 一等参数 + 路径结构重构）——与"不改路径结构"约束冲突，不采纳；"title 作为 CLI 参数"转 sug 登记到后续批次。
