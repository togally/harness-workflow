# 变更简报：chg-05 对人文档落盘校验机制

## 变更名

- 提供 `harness validate --human-docs` 子命令，校验 req / bugfix 的 6 份对人文档是否真实落盘。

## 解决什么问题

- 对应 AC-09：当前契约只规定"对人文档必须产出"，但没有可跑的校验入口，AC-11 的"首次完整示范"没有反向验证手段。

## 怎么做

- 把 `stage-role.md` 契约 3 映射表硬编码为 `HUMAN_DOC_CONTRACT`。
- 实现 `validate_human_docs(target_id)`，逐条校验 req / bugfix / change 层对人文档是否存在。
- CLI 加 `harness validate --human-docs [--requirement <id> | --bugfix <id>]`，缺失非零退出。
- `acceptance.md` SOP 与 scaffold_v2 同步补"验收开始前必跑"条目。

## 影响范围

- 代码：`src/harness_workflow/validate_human_docs.py`（新文件）、`src/harness_workflow/cli.py`、`.workflow/context/roles/acceptance.md`、`scaffold_v2/roles/acceptance.md`。
- 用户路径：`harness validate --human-docs`。
- 角色 SOP：acceptance 验收开始前的硬门禁。

## 预期验证

- `harness validate --human-docs --requirement req-28` 随着推进从"只含需求摘要"逐步变为"全 ok"。
- 构造 fixture 的失败用例时退出码非 0 且路径报错清晰。
- `acceptance.md` 可 grep 到新 SOP 条目。
- `tests/test_validate_human_docs.py` 全绿。
